# v1.4 功能实现清单

## 当前阶段目标

v1.4 重点补齐通知、游客报名、组队大厅空态、组局可见性和活动订票体验，同时继续处理线上预览环境登录后的明显性能问题。

当前主线：

```text
更清楚地收到通知 -> 游客也能先报名 -> 新用户知道下一步做什么 -> 组局可见性和参与关系更明确 -> 需要订票的活动能直接跳转购票
```

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull origin dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* -> dev
```

不要把通知中心、游客报名、性能优化、组队大厅空态和组局参与关系合并到同一个分支里。每个分支只解决一个明确问题，方便测试、回滚和上线验证。

### 2. 游客报名与注册后自动关联

建议分支：

```text
feature/guest-signup-account-linking
```

需求理解：

当前活动或组队报名强依赖登录，会挡住一部分临时用户；微信 WebView 中登录还需要跳转浏览器，流失更明显。v1.4 需要允许用户先用 H5 游客身份报名，之后再通过邮箱或微信号把游客报名记录自动归属到正式账户。这个能力应该让游客能正常参加活动，也让后续注册、登录、补充微信号的用户找回历史报名记录。

小功能：

- [x] 未登录用户可以用游客身份报名活动或组队，不强制跳转登录
- [x] 游客报名表单需要收集名字 / 昵称、电话号码、邮箱、微信号
- [x] 电话、邮箱、微信号至少填写一种联系方式；名字 / 昵称必填
- [x] 表单提示：邮箱或微信号用于之后注册、登录或补充资料时找回报名记录
- [x] 微信 WebView 中优先保留 H5 游客报名路径，避免因为登录跳转浏览器造成流失
- [x] 新增游客报名记录模型，保存活动 / 组队、展示昵称、电话、邮箱、微信号、报名留言、创建时间、来源环境
- [x] 游客报名成功后，在活动参与人区域展示游客的名字 / 昵称
- [x] 游客头像用名字 / 昵称的第一个字符生成，支持中文首字，并用稳定哈希生成不同底色，类似 Google 字母头像
- [x] 用户资料新增微信号字段，允许用户填写和修改
- [x] 用户注册、登录或资料更新后，如果已验证账户邮箱与游客报名邮箱一致，自动绑定对应游客报名记录
- [x] 用户填写或修改微信号后，如果与游客报名微信号一致，自动绑定对应游客报名记录
- [x] 已有账户用户如果曾经用同一邮箱游客报名，登录后也能自动找回并归属报名记录
- [x] 绑定成功后，游客报名记录转为正式用户参与记录，活动详情和个人报名记录都能看到
- [x] 绑定过程需要幂等，重复登录、重复修改微信号不会创建重复参与记录
- [x] 如果游客报名对应活动已经结束、取消或报名状态变化，绑定时需要保留历史状态，不错误重开报名
- [x] 对联系方式做必要脱敏展示，避免在活动详情中公开手机号、邮箱或微信号
- [x] 防止不同用户通过随意填写联系方式错误认领他人记录

验收标准：

- 未登录用户可以在 H5 和微信 WebView 中完成报名提交
- 游客报名后，活动参与人区域能看到游客昵称头像，但不会暴露联系方式
- 注册或登录后，匹配已验证账户邮箱的游客记录能自动归属到正式账户
- 用户填写或修改微信号后，匹配微信号的游客记录能自动归属到正式账户
- 正式用户能在自己的报名记录中看到被自动归属的历史游客报名
- 信息不一致时不会误关联
- 同一游客记录不会被多个用户重复认领
- 重复提交、重复关联和恶意认领都有防护
- 微信 WebView 中不再因为必须登录才能报名而阻断用户

实现注意事项：

- 邮箱自动绑定依赖 Clerk 已验证邮箱，避免只靠未验证输入认领他人报名
- 微信号属于用户手动填写信息，已做大小写、空格和最小长度规范化；如果游客记录同时有邮箱和微信，微信匹配还需要已验证邮箱一致，避免被别人只填微信号抢走
- 电话号码可以作为游客联系方式保存；是否用于自动绑定取决于正式用户资料是否已有可信手机号字段
- 游客记录和正式参与记录之间需要保留可追溯关系，方便排查误绑定
- 自动绑定不应该覆盖用户已有参与状态，也不应该绕过活动容量、审批、取消等业务规则
- 对游客报名接口增加节流、防重复提交和基础垃圾数据防护；当前使用来源 hash 指纹限制短时间内同活动和全站提交频率，不保存原始 IP

### 2.1 微信号绑定入口独立化与归属隐私确认

建议分支：

```text
feature/wechat-binding-account-menu
```

需求理解：

微信号用于把游客报名记录和正式账户关联，但它不应该挤在“编辑昵称”的表单里。微信号属于账号绑定信息，应放在用户头像下拉菜单和账号相关入口中展示：未绑定时使用灰色微信标，提示用户可以绑定；绑定成功后变为正常微信绿色标。点击后打开独立的绑定 / 修改微信号表单，并解释为什么需要微信号：用于找回和关联曾经通过微信号提交的游客报名记录，不会在活动详情公开展示。

同时需要确认并保持活动 / 组局发起归属不依赖昵称。归属、权限判断、编辑权限、发起人身份、报名归属都必须使用稳定的 `UserProfile.id` / `organizerId` / `clerkUserId` 等内部标识，不能用昵称、展示名或微信号作为权限依据，避免昵称变更或重名带来的隐私和安全问题。

小功能：

- [x] 从“编辑昵称”面板中移除微信号输入，不把昵称编辑和微信绑定混在一起
- [x] 头像下拉头部的好友号右侧增加微信状态标：未绑定为灰色，已绑定为微信绿色
- [x] 点击灰色微信标或下拉栏“绑定微信号”打开独立绑定表单
- [x] 已绑定用户下拉栏展示“修改微信号”，点击后打开同一套修改表单
- [x] 绑定 / 修改表单需要说明微信号用途：用于关联游客报名和找回历史报名，不公开给其他用户
- [x] 绑定成功后头像下拉头部微信标从灰色变为绿色
- [x] 绑定成功后触发已有游客报名关联逻辑，并给用户明确成功反馈
- [x] 微信号保存、修改和自动关联需要保持幂等，避免重复生成参与记录
- [x] 微信号不在活动详情、组队详情、参与人列表、通知或公开页面中明文展示
- [x] 审计活动 / 组局创建、编辑、取消、报名审批等权限判断，确认均使用内部 profile id，不使用昵称
- [x] 列表筛选中的“我发起的”“好友发起”“好友参加”继续使用 `organizerId`、好友 id、参与记录 id 关系判断
- [x] 如发现任何通过 nickname / displayName 判断归属或权限的逻辑，需要改为内部 id

当前代码确认：

- [x] `Activity.organizerId` 是 `UserProfile.id` 外键，不是昵称
- [x] 创建组局时写入 `organizerId: profile.id`
- [x] 组队大厅“我发起的 / 好友发起”等筛选使用 `organizerId` 和好友 profile id 判断
- [x] 当前检索到的昵称主要用于展示发起人、参与人和头像首字母，不作为归属判断依据

验收标准：

- 用户不会在编辑昵称时看到微信号输入
- 用户能在头像下拉中清楚看到微信号是否已绑定
- 用户知道绑定微信号的目的，并且不会误以为微信号会公开展示
- 绑定 / 修改微信号后游客报名归属能正常同步
- 活动发起归属和权限判断不依赖昵称、展示名或微信号
- `npm run typecheck` 通过

### 3.1 登录后回到原目标页面

建议分支：

```text
feature/auth-return-to-intended-page
```

需求理解：

当前用户从活动详情、组队详情、报名、收藏、管理等需要登录的入口进入登录流程后，完成登录有时会被带回首页。这会打断用户原本想完成的动作，尤其是在微信分享、活动详情页和组队大厅场景中体验很差。登录流程应该记住用户原本想去的页面或动作上下文，登录 / 注册完成后优先回到这个安全的站内地址。

小功能：

- [x] 未登录用户访问需要登录的页面或点击需要登录的操作时，记录原目标地址
- [x] 登录和注册完成后优先跳回原目标页面，而不是固定回首页
- [x] 保留当前 locale、query 参数和必要 hash，避免语言和筛选状态丢失
- [x] 支持活动详情、公共活动详情、组队详情、组队大厅、消息、通知和个人页等常见入口
- [x] 对 redirect URL 做站内白名单校验，防止 open redirect
- [x] 没有安全回跳地址时再回到默认首页或当前 locale 首页
- [x] 微信 WebView、普通移动浏览器和桌面端登录路径保持一致
- [x] 登录后如果原操作仍需用户点击确认，不要自动执行高风险动作，只回到上下文

验收标准：

- 从详情页点击登录后，登录完成回到同一个详情页
- 从报名、收藏、管理等入口触发登录后，能回到原页面继续操作
- 不同 locale 下登录后不会跳错语言
- 恶意外部 redirect 参数不会被执行
- `npm run typecheck` 通过

### 3.2 组队大厅与活动页分页加载优化

建议分支：

```text
feature/lobby-activity-pagination-performance
```

需求理解：

组队大厅第一次加载明显慢，但翻页很快，说明首屏可能承担了过多数据准备、分区统计或后续页预热。需要把首屏目标改成“只加载当前页必要数据”，后续页在用户翻页时按需加载；如果要预加载，也只能在空闲时低优先级加载下一页，不能阻塞用户当前页面。这个优化要同时考虑筛选器、分类计数、登录态好友筛选和活动页是否存在类似问题。

小功能：

- [x] 分析组队大厅首屏、翻页、筛选切换的真实耗时和查询数量
- [x] 默认列表只加载当前页所需卡片，不一次性加载全部页数据
- [x] 筛选器计数使用轻量统计或按需加载，不阻塞当前页卡片展示
- [x] 翻页时按页请求数据，保留已访问页缓存，避免重复等待
- [x] 只在浏览器空闲时预取下一页 / 上一页，且用户点击翻页时优先处理当前请求
- [x] 收藏、好友发起、好友参加等登录态分区不能因为统计或预热拖慢默认列表
- [x] 检查活动页是否也存在一次性加载全部分页数据的问题，并按相同策略优化
- [x] 加载态、错误态和空态在分页、筛选切换时都要稳定，不出现长时间空白

实现说明：

- 组队大厅默认 feed 改为服务端按页返回，首屏只取当前页卡片和轻量总数。
- 新增 `/api/lobby/feed` 支持按状态和页码加载大厅列表，并在前端缓存已访问页。
- 移动端探索滑卡改为 `/api/lobby/swipe` 空闲时加载，不阻塞大厅首屏。
- 收藏、好友发起、好友参加继续作为延迟分区加载，不参与默认列表首屏阻塞。
- 活动页公共活动推荐列表改为先 count 再读取受控候选集，避免默认推荐排序全量读取。

验收标准：

- 组队大厅首屏不会因为后续页数据阻塞
- 翻页时只加载目标页必要数据，并有明确 loading 反馈
- 筛选条件变化后计数和列表结果一致
- 活动页如果存在同类问题，也完成同策略优化
- `npm run typecheck` 通过

### 6.1 长时段组局发起前二次确认

补充：微信分享链接时，应尽量以卡片形式展示活动 / 组队标题、摘要和缩略图。该能力主要依赖页面 Open Graph / Twitter Card 元信息，微信客户端最终展示样式不可由网页完全强制控制，但可以通过动态 metadata 和稳定的分享图显著提升展示成功率。

#### 微信分享卡片设计说明

目标不是在网页中“强制生成微信卡片”，而是让微信在用户分享链接时更容易抓取到正确的标题、摘要和缩略图。用户最终看到的卡片样式由微信客户端决定，网站侧能做的是提供稳定、公开、可抓取的分享页面和元信息。

微信卡片形成流程：

1. 用户在 Next Fun 的活动详情页、公共活动详情页或组队详情页点击分享入口。
2. 用户复制链接、使用系统分享，或在微信内置浏览器中通过微信菜单分享给好友 / 群。
3. 微信客户端或微信抓取服务访问这个公开 URL。
4. 微信读取页面 `<head>` 中的 `title`、`description`、`og:title`、`og:description`、`og:image` 等信息。
5. 微信根据抓取结果生成聊天中的链接卡片。
6. 好友点击卡片后进入对应详情页，未登录也能先看到活动 / 组队的基础信息，再决定报名、游客报名、登录或发起组队。

一期方案：先做动态页面 metadata，不接微信 JS-SDK。

- 适用场景：用户复制链接到微信，或使用浏览器系统分享。
- 优点：成本低，不需要微信开放平台配置；对微信、Telegram、Facebook、X、iMessage 等平台都有效。
- 局限：不能完全控制微信最终展示样式；微信缓存可能导致新标题或新图片不是即时更新。
- 实施重点：让公开详情页在 SSR 阶段稳定输出正确 metadata，且图片 URL 是公网 HTTPS 绝对地址。

二期可选方案：接入微信 JS-SDK 分享能力。

- 适用场景：用户已经在微信内置浏览器打开页面，并点击微信右上角菜单分享。
- 可能收益：在微信 WebView 内能更明确地设置分享给朋友、分享到朋友圈的标题、描述、链接和图片。
- 前置条件：需要可用的微信公众号 / 开放平台配置、JS 安全域名、签名接口、域名备案/认证等微信侧要求。
- 风险和成本：配置链路更长，微信规则变化时维护成本更高；仍然不能保证所有微信版本都按预期展示。
- 建议：v1.4 先完成一期，等真实分享卡片数据稳定后，再评估是否值得做二期。

需要覆盖的页面：

- 公共活动详情页：`/[locale]/public-events/[publicEventId]`
- 组队详情页：`/[locale]/activities/[activityId]`
- 活动详情页中由真实组队生成的分享入口，也应该指向公开可访问的详情页
- 如果未来有独立的短链或分享落地页，需要复用同一套 metadata 生成逻辑

metadata 字段设计：

- `title`：优先使用活动 / 组队标题，长度控制在 20 到 40 个中文字符附近，过长时截断并保留重点。
- `description`：使用简介首句 + 时间 + 地点组合，避免塞入完整长文案。
- `openGraph.title`：与页面标题一致，便于微信和其他平台识别。
- `openGraph.description`：与摘要一致，保证复制链接后卡片不是空摘要。
- `openGraph.url`：使用当前详情页公网 HTTPS 绝对 URL，不使用 localhost。
- `openGraph.siteName`：使用 `Next Fun`。
- `openGraph.images`：优先使用详情页封面图；没有封面时使用默认品牌分享图。
- `twitter.card`：使用 `summary_large_image`，让其他平台也能展示大图卡片。
- `alternates.canonical`：指向当前 locale 的正式详情页，减少重复 URL 引起的抓取不稳定。

分享图设计：

- 默认尺寸建议使用 `1200x630`，兼容 Open Graph 大图卡片。
- 图片必须是公网可访问的 HTTPS 绝对地址，不能是相对路径、localhost、内网地址或需要登录的地址。
- 图片内容应该包含活动 / 组队核心视觉，不要只用纯 logo。
- 有封面图时优先使用封面图；封面图尺寸过小、加载失败或非 HTTPS 时回退默认品牌图。
- 默认品牌图固定使用 `apps/web/public/logo.png`，对应公网路径 `/logo.png`；以后品牌图更新也只替换同路径同文件名的 png，不改代码里的默认路径。
- 对于已知容易防盗链或拒绝外部抓取的活动封面，分享卡片应使用站内公开代理地址，避免微信直接抓原站图片失败。
- 如果后续生成动态 OG 图，建议包含：新 logo、标题、时间、地点、费用 / 报名状态、二维码或短链接。
- 动态 OG 图接口必须无需登录访问，并设置合理缓存，避免每次微信抓取都触发重计算。

提高微信卡片成功率的重要事项：

- 分享链接必须是生产或预览环境的 HTTPS URL，本地 `localhost:3000` 不会被微信正常抓取。
- 详情页基础信息不能依赖登录态，否则微信抓取时可能只得到登录页或空页面。
- 页面 SSR 不能因为用户未登录、数据库字段缺失、天气接口失败、收藏状态失败而抛 500。
- `og:image` 不能指向会 302 到登录页、会过期、会防盗链失败或返回非图片内容的地址。
- 站内封面代理返回图片时应使用公开缓存，降低微信抓取超时和重复请求失败概率。
- 分享图片最好避免太大，减少微信抓取超时概率。
- 标题和摘要不要依赖客户端渲染后再补；微信抓取通常只看服务端返回的 HTML。
- 微信可能缓存同一个 URL 的卡片内容。发布后如果要测试新卡片，可以换一条真实详情页、追加稳定版本参数，或等待缓存刷新。
- 不要把长网址直接作为卡片标题或按钮文案；用户看到的应该是活动内容，而不是技术 URL。
- 多语言页面需要按 URL locale 输出对应文案；没有翻译时可以显示原文，但按钮和固定字段必须本地化。
- 分享入口上的“复制链接”和“系统分享”都应使用同一个 canonical URL，避免同一活动产生多个卡片版本。

用户操作流程：

1. 用户进入活动详情页或组队详情页。
2. 用户看到“活动分享”或“组队分享”入口。
3. 用户点击“复制链接”或系统分享按钮。
4. 用户打开微信聊天窗口，把链接发给好友或群。
5. 微信生成包含标题、摘要和缩略图的链接卡片。
6. 好友点击微信卡片，进入 Next Fun 对应详情页。
7. 好友未登录时也能先看基础信息；需要报名时可以选择登录报名或游客报名。
8. 如果微信暂时只显示普通链接，用户仍然可以发送链接；卡片失败不应阻断分享流程。

测试流程：

1. 部署到预览环境，拿到 HTTPS 预览地址。
2. 打开公共活动详情页、组队详情页，查看页面源码中是否存在完整 `og:title`、`og:description`、`og:image`、`og:url`。
3. 用未登录浏览器访问分享 URL，确认不会跳登录页、不会 500。
