# v1.4 功能实现清单

## 当前阶段目标

v1.4 重点补齐通知、游客报名、组队大厅空态和组局可见性体验，同时继续处理线上预览环境登录后的明显性能问题。

当前主线：

```text
更清楚地收到通知 -> 游客也能先报名 -> 新用户知道下一步做什么 -> 组局可见性和参与关系更明确
```

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull origin dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* -> dev
```

不要把通知中心、游客报名、性能优化、组队大厅空态和组局参与关系合并到同一个分支里。每个分支只解决一个明确问题，方便测试、回滚和上线验证。

## P0：v1.4 必须优先解决的核心体验

### 1. 通知中心页面优化

建议分支：

```text
feature/notification-center-polish
```

小功能：

- [ ] 梳理通知中心的信息层级，区分未读、已读和不同通知类型
- [ ] 优化通知列表的空态、加载态和错误态
- [ ] 通知卡片展示清晰的触发人、动作、关联活动或组队
- [ ] 通知点击后跳转到对应活动、组队、消息或好友请求上下文
- [ ] 已读状态和顶部通知数字保持同步
- [ ] 移动端通知列表不被导航栏遮挡，长文案不挤压操作区

验收标准：

- 用户能快速判断哪些通知需要处理
- 点击通知后能进入正确上下文
- 已读后通知数字能合理下降
- 通知中心没有长时间空白或无反馈等待

### 2. 游客报名与注册后自动关联

建议分支：

```text
feature/guest-signup-account-linking
```

小功能：

- [ ] 未登录用户可以用游客身份报名活动或组队
- [ ] 游客报名至少支持填写电话、邮箱或微信号中的一种联系方式
- [ ] 微信环境下尽量自动关联微信身份或微信号线索
- [ ] 游客报名记录需要保留来源活动、组队、联系方式和创建时间
- [ ] 用户注册或登录后，如果邮箱、电话或微信号一致，自动关联历史游客报名记录
- [ ] 关联成功后，用户可以在自己的报名记录中看到原游客报名
- [ ] 防止不同用户通过随意填写联系方式错误认领他人记录
- [ ] 对联系方式做必要脱敏展示，避免暴露隐私

验收标准：

- 未登录用户可以完成报名提交
- 注册或登录后，匹配个人信息的游客记录能自动归属到正式账户
- 信息不一致时不会误关联
- 重复提交、重复关联和恶意认领都有防护

### 3. 网页性能优化：登录后预览环境加载慢

建议分支：

```text
feature/preview-auth-performance
```

小功能：

- [ ] 记录线上预览环境登录前、登录后关键页面首屏加载耗时
- [ ] 排查登录后是否触发过多用户态查询、串行查询或全局 layout 查询
- [ ] 检查通知、好友、头像菜单、权限判断等全局数据是否拖慢每个页面
- [ ] 将可并行的数据读取改为并行读取
- [ ] 减少登录态公共布局中不必要的阻塞请求
- [ ] 对非强实时数据使用缓存、延迟加载或客户端渐进加载
- [ ] 给超过预期的用户态数据加载增加稳定 loading 或 shimmer
- [ ] 输出性能结论，明确慢点来自 Preview 冷启动、数据库查询、认证链路还是页面实现

验收标准：

- 登录后页面不应频繁出现超过 10 秒的无反馈等待
- 可以明确定位主要耗时来源
- Preview 环境冷启动时也有明确加载反馈
- 优化后关键页面有前后耗时对比
- `npm run typecheck` 通过

## P1：组队大厅和组局关系体验

### 4. 组队大厅新用户空态

建议分支：

```text
feature/lobby-new-user-empty-state
```

小功能：

- [ ] 当用户没有相关组队数据时展示引导空态
- [ ] 空态文案覆盖“先加好友”“我来组局”“去发现活动”三个方向
- [ ] 空态中的按钮能跳转到添加好友、发起组局和活动发现页
- [ ] 筛选条件为 0 条结果时，展示和新用户首次进入不同的空态
- [ ] 空态卡片在移动端和桌面端都不挤压筛选区
- [ ] 保持与当前组队大厅视觉风格一致

参考文案：

```text
你的组队大厅还空着
公园？逛展？还是看电影？先组个局，去逛逛活动，再加几个好友吧。
```

验收标准：

- 新用户进入组队大厅时知道下一步可以做什么
- 三个引导入口都能正常跳转
- 筛选无结果和真正无数据的状态可区分
- 页面不再只显示空列表或 0 计数

### 5. 开放局 / 私人局显示

建议分支：

```text
feature/open-private-crew-display
```

小功能：

- [ ] 组队大厅列表明确展示开放局和私人局标签
- [ ] 筛选项中的“开放”计数和列表展示保持一致
- [ ] 组队详情页展示当前局的可见范围
- [ ] 私人局只对符合权限的用户显示
- [ ] 好友可见、陌生人不可见的规则需要在列表、详情和搜索入口保持一致
- [ ] 对不可见或已失效的私人局给出合理兜底，不暴露敏感信息

验收标准：

- 用户能从列表直接判断这是开放局还是私人局
- 非好友不能在公开列表看到私人局
- 好友可以正常看到私人局
- 计数、筛选结果和实际列表一致

### 6. 自己发起的组局增加“是否自己参与”按钮

建议分支：

```text
feature/organizer-self-participation-toggle
```

小功能：

- [ ] 发起组局时增加“我也参与”开关
- [ ] 编辑自己发起的组局时可以修改是否自己参与
- [ ] 默认值需要符合当前产品预期，并在 UI 中表达清楚
- [ ] 参与人数、剩余名额和报名列表需要正确计算发起人是否参与
- [ ] 发起人不参与时，不应占用报名名额
- [ ] 发起人参与时，详情页和报名列表需要展示发起人的参与状态
- [ ] 防止发起人重复作为普通报名用户占用两次名额

验收标准：

- 发起人可以明确选择自己是否参与组局
- 名额计算和报名列表与选择状态一致
- 编辑后详情页、列表卡片和筛选计数同步更新
- `npm run typecheck` 通过
