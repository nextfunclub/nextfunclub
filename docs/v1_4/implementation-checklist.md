# v1.4 功能实现清单

## 当前阶段目标

v1.4 重点补齐通知、游客报名、组队大厅空态、组局可见性和活动订票体验，同时继续处理线上预览环境登录后的明显性能问题。

当前主线：

```text
更清楚地收到通知 -> 游客也能先报名 -> 新用户知道下一步做什么 -> 组局可见性和参与关系更明确 -> 需要订票的活动能直接跳转购票
```

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull origin dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* -> dev
```

不要把通知中心、游客报名、性能优化、组队大厅空态和组局参与关系合并到同一个分支里。每个分支只解决一个明确问题，方便测试、回滚和上线验证。

### 2. 游客报名与注册后自动关联

建议分支：

```text
feature/guest-signup-account-linking
```

需求理解：

当前活动或组队报名强依赖登录，会挡住一部分临时用户；微信 WebView 中登录还需要跳转浏览器，流失更明显。v1.4 需要允许用户先用 H5 游客身份报名，之后再通过邮箱或微信号把游客报名记录自动归属到正式账户。这个能力应该让游客能正常参加活动，也让后续注册、登录、补充微信号的用户找回历史报名记录。

小功能：

- [x] 未登录用户可以用游客身份报名活动或组队，不强制跳转登录
- [x] 游客报名表单需要收集名字 / 昵称、电话号码、邮箱、微信号
- [x] 电话、邮箱、微信号至少填写一种联系方式；名字 / 昵称必填
- [x] 表单提示：邮箱或微信号用于之后注册、登录或补充资料时找回报名记录
- [x] 微信 WebView 中优先保留 H5 游客报名路径，避免因为登录跳转浏览器造成流失
- [x] 新增游客报名记录模型，保存活动 / 组队、展示昵称、电话、邮箱、微信号、报名留言、创建时间、来源环境
- [x] 游客报名成功后，在活动参与人区域展示游客的名字 / 昵称
- [x] 游客头像用名字 / 昵称的第一个字符生成，支持中文首字，并用稳定哈希生成不同底色，类似 Google 字母头像
- [x] 用户资料新增微信号字段，允许用户填写和修改
- [x] 用户注册、登录或资料更新后，如果已验证账户邮箱与游客报名邮箱一致，自动绑定对应游客报名记录
- [x] 用户填写或修改微信号后，如果与游客报名微信号一致，自动绑定对应游客报名记录
- [x] 已有账户用户如果曾经用同一邮箱游客报名，登录后也能自动找回并归属报名记录
- [x] 绑定成功后，游客报名记录转为正式用户参与记录，活动详情和个人报名记录都能看到
- [x] 绑定过程需要幂等，重复登录、重复修改微信号不会创建重复参与记录
- [x] 如果游客报名对应活动已经结束、取消或报名状态变化，绑定时需要保留历史状态，不错误重开报名
- [x] 对联系方式做必要脱敏展示，避免在活动详情中公开手机号、邮箱或微信号
- [x] 防止不同用户通过随意填写联系方式错误认领他人记录

验收标准：

- 未登录用户可以在 H5 和微信 WebView 中完成报名提交
- 游客报名后，活动参与人区域能看到游客昵称头像，但不会暴露联系方式
- 注册或登录后，匹配已验证账户邮箱的游客记录能自动归属到正式账户
- 用户填写或修改微信号后，匹配微信号的游客记录能自动归属到正式账户
- 正式用户能在自己的报名记录中看到被自动归属的历史游客报名
- 信息不一致时不会误关联
- 同一游客记录不会被多个用户重复认领
- 重复提交、重复关联和恶意认领都有防护
- 微信 WebView 中不再因为必须登录才能报名而阻断用户

实现注意事项：

- 邮箱自动绑定依赖 Clerk 已验证邮箱，避免只靠未验证输入认领他人报名
- 微信号属于用户手动填写信息，已做大小写、空格和最小长度规范化；如果游客记录同时有邮箱和微信，微信匹配还需要已验证邮箱一致，避免被别人只填微信号抢走
- 电话号码可以作为游客联系方式保存；是否用于自动绑定取决于正式用户资料是否已有可信手机号字段
- 游客记录和正式参与记录之间需要保留可追溯关系，方便排查误绑定
- 自动绑定不应该覆盖用户已有参与状态，也不应该绕过活动容量、审批、取消等业务规则
- 对游客报名接口增加节流、防重复提交和基础垃圾数据防护；当前使用来源 hash 指纹限制短时间内同活动和全站提交频率，不保存原始 IP

### 10. 旧活动订票链接回填

建议分支：

```text
feature/backfill-ticket-links
```

目标：

对已经入库的 Open Data 活动进行一次性回填，把历史 `sourcePayload` 里的订票或预约链接补到新增的结构化字段里，避免只有后续导入的新活动才有“立即抢票”。

小功能：

- [ ] 新增一次性回填脚本，默认 dry-run，只输出会更新的数量和样例
- [ ] 支持 `--write` 或明确参数后才真实写库
- [ ] 优先回填 `PublicEvent.ticketUrl` / `PublicEvent.ticketLabel`
- [ ] 如仍存在旧 `Activity` 形式的 Open Data 活动，再回填 `Activity.ticketUrl` / `Activity.ticketLabel`
- [ ] 从 `sourcePayload.access_link` 读取订票或预约链接
- [ ] 从 `sourcePayload.access_link_text` 读取按钮文案，但需要复用订票文案过滤规则
- [ ] 回填后输出 created / updated / skipped / invalidUrl 等统计

注意事项：

- 只处理 `source` 属于 `paris-opendata` / `paris-opendata:que-faire-a-paris` 的历史数据
- 只补 `ticketUrl` 为空的记录，不能覆盖人工编辑过的订票链接
- `access_link` 必须是有效 `http` 或 `https` URL，其他值跳过
- `access_link_text` 如果是 URL、过长、或通用法语文案如 `Réservation`，不要写入 `ticketLabel`
- 回填前先在 preview 数据库 dry-run，并保留样例 title / id / ticketUrl 供人工确认
- 生产环境执行前需要确认数据库备份或可回滚方案

验收标准：

- dry-run 能明确显示将回填多少条、跳过多少条、无效链接多少条
- `--write` 执行后，历史 Open Data 活动详情页能出现“立即抢票”
- 人工配置过订票链接的活动不会被覆盖
- 回填脚本可重复执行，重复执行不会产生额外副作用
