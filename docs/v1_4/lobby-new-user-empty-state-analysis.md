# 组队大厅新用户空态分析与讨论

本文只做产品和 UX 探索，不包含实现方案。目标是讨论：当新用户没有好友、没有报名、没有发起组局时，组队大厅如何仍然显得有内容、有方向、有行动价值。

## 背景

组队大厅在桌面端是一个重要入口，在移动端更接近首页。当前截图里的状态是：

- 顶部筛选全部为 `0`。
- 主标题直接告诉用户「你的组队大厅还空着」。
- 页面核心内容是三个跳转卡片：加好友、我来组局、去发现活动。
- 用户如果不主动跳走，当前页面没有可浏览内容，也没有可立即参与的对象。

这对普通功能页可以接受，但对移动端首页级入口不够。首页级页面不能只是一个「空数据提醒 + 跳转集合」，它应该在用户没有数据时仍然提供发现、启动、学习和参与。

## 核心判断

当前问题不是单纯的 empty state 文案问题，而是冷启动产品结构问题。

组队大厅现在更像一个「关系型动态页」：

- 我发起的
- 我参加的
- 我收藏的
- 好友发起
- 好友参加

但新用户刚进来时没有关系、没有历史、没有参与记录，所以这些分区天然为空。如果页面只依赖这些数据，冷启动必然失败。

更准确的定位应该是：

> 组队大厅 = 我的组队动态 + 可加入的公开机会 + 帮我发起第一条组局的行动面板。

也就是说，新用户阶段不能只展示「我的内容」，必须补上「平台可参与内容」和「引导我产生第一条内容」。

## 当前空态的问题

### 1. 页面第一眼传达的是「这里没人」

筛选按钮全部显示 `0`，再配合「还空着」的标题，会强化产品冷清感。用户不会理解这是因为自己刚注册，而会倾向于认为平台没有人、没有活动、没有价值。

### 2. 三张卡片本质都是离开当前页面

当前三个方向都合理，但都不是组队大厅里的内容：

- 加好友：跳去关系建设。
- 我来组局：跳去创建流程。
- 去发现活动：跳去活动市场。

它们是行动入口，不是内容填充。页面仍然空。

### 3. 没有利用已有公开活动和公开局

即使用户没有个人数据，平台依然可能有：

- 公开活动
- 公开组局
- 即将开始的活动
- 可围绕活动发起的组队机会
- 最近活跃的同城用户

这些都可以成为大厅内容，不应该等用户跳到活动页才看见。

### 4. 不适合移动端首页角色

移动端用户打开 app 式网页时，底部第一个入口就是组队大厅。这个页面承担的是「今天我能和谁一起做什么」的启动任务，而不是单纯告诉用户「你还没有数据」。

## 从之前讨论提炼出的方向

用户提供的旧讨论里，几个竞品方向可以转化成产品原则：

- Meetup：先让用户发现同城活动，而不是先要求用户有好友。
- Bumble BFF：用人和兴趣降低社交冷启动门槛。
- Timeleft：平台可以给用户安排或推荐一个可参与的社交场景。
- Partiful：活动本身可以成为社交关系入口。
- Rodeo：把「想去哪里」变成「和谁一起去」的计划。
- LinkedIn：用资料完成度、关系推荐、与我相关动态，把空首页变成下一步行动面板。

对 Next Fun 来说，最贴近的组合不是单纯信息流，而是：

> Meetup 的活动发现 + Bumble BFF 的找搭子 + Partiful 的活动社交化 + LinkedIn 的新手任务和关系推荐。

## 设计原则

### 原则 1：不要做假热闹

不建议放虚假的热门动态、虚构用户、虚构组队。短期看起来丰满，长期会伤害信任。

更好的方式是用真实数据和可启动模板：

- 真实公开活动
- 真实公开组局
- 真实可报名活动
- 真实近期活跃用户
- 可发起的组队模板

### 原则 2：新用户看到的是「下一步」，不是「暂无数据」

空态应该回答：

- 我现在可以做什么？
- 我可以和谁产生联系？
- 我可以围绕什么活动发起第一条组局？
- 我为什么要留在这个页面继续看？

### 原则 3：组队大厅不是活动页副本

活动页是完整市场，负责浏览全部活动。

组队大厅应该更偏行动：

- 找人一起去
- 加入已有局
- 从某个活动发起组局
- 看朋友或同城的人在做什么

同一个活动在活动页里是「活动卡」，在组队大厅里应该是「可以一起去的机会」。

### 原则 4：新用户和老用户应渐进切换

不应该做两个完全割裂的首页，而应该根据用户数据量渐进变化。

新用户：

- 推荐活动
- 推荐组局机会
- 推荐搭子
- 新手任务
- 快速发布找搭子

活跃用户：

- 我的下一场活动
- 我的组局状态
- 好友动态
- 待确认邀请
- 与我相关的推荐

## 建议的信息架构

### 新用户大厅首屏

建议首屏不要再以「你的大厅还空着」开头，而是改成一个行动型开场：

```text
今天想找谁一起出门？

先选一个方向，Next Fun 会帮你把活动、组队和可能合拍的人放到一起。

[找活动一起去] [发起一个局] [看看公开局]
```

这里的重点是让用户感觉页面正在帮他启动，而不是宣布没有内容。

### 模块 1：正在开放的组队机会

优先展示真实公开局。如果公开局不足，可以展示公开活动生成的「可组队机会」。

例子：

```text
这些活动适合找人一起去

巴黎桌游夜
已有 2 个组局 / 还可以发起新局

周末咖啡聊天
还没有组局，做第一个发起的人
```

这类内容的价值是：即使没有真实组队，也可以把公开活动转化为「组队种子」。

### 模块 2：一键发布找搭子

这是组队大厅的核心能力，不应该藏在空态里的一个卡片。

建议做成轻量 prompt：

```text
你今天想找什么搭子？

[吃饭] [咖啡] [运动] [展览] [学习] [周末散步]

[发布找搭子]
```

如果后续实现，可以先生成带默认标题和分类的创建页草稿，而不是让用户从空白表单开始。

### 模块 3：适合新用户的活动

展示 3 到 5 个适合冷启动的活动，不是完整活动页。

筛选标准可以优先考虑：

- 即将开始
- 免费或低门槛
- 地点明确
- 适合多人参与
- 有公开局或适合组局
- 标题和图片质量较高

按钮文案不要只有「查看活动」，应更贴近组队：

- 找人一起去
- 加入已有局
- 我感兴趣

### 模块 4：可能合拍的人

如果已有用户资料足够，可以展示同城、同兴趣、最近活跃的人。这个模块要谨慎，隐私和质量比数量重要。

最低可用版本可以不直接推荐用户，而是推荐「人群入口」：

```text
想认识哪类搭子？

[同城周末搭子] [展览搭子] [运动搭子] [语言练习搭子]
```

点击后可以进入搜索或创建组局，而不是马上暴露用户列表。

### 模块 5：资料完成度

当用户没有足够数据时，资料完成度是合理的填充内容，但它不能占据首屏中心。

建议放在中后段：

```text
让别人更容易加入你的局

资料完成度 40%
✓ 选择城市
✓ 添加兴趣
□ 上传头像
□ 写一句自我介绍
□ 选择想找的搭子类型
```

它的目的不是装饰，而是提高后续匹配和推荐质量。

## 页面状态分层

### 0 数据新用户

特征：

- 没有好友
- 没有报名
- 没有发起
- 没有收藏

展示重点：

- 新手启动面板
- 公开活动组队机会
- 一键找搭子 prompt
- 适合新用户活动
- 资料完成度

### 少数据用户

特征：

- 有 1 到 2 个参与记录
- 可能没有好友
- 公开局不多

展示重点：

- 我的下一步
- 我的组局或报名
- 基于已参与活动的推荐
- 与当前活动相关的组队机会
- 继续补充同城和兴趣内容

### 活跃用户

特征：

- 有好友
- 有报名
- 有发起
- 有互动

展示重点：

- 我的下一场活动
- 组队进度
- 好友动态
- 待处理邀请
- 与我相关的活动和人

## 推荐的移动端顺序

移动端组队大厅建议顺序：

1. 行动型标题和快速入口
2. 我的下一步
3. 正在开放的组队机会
4. 适合你一起去的活动
5. 一键发布找搭子
6. 可能合拍的人或人群入口
7. 资料完成度

如果用户已经有真实组队数据，则「我的下一步」和「我的组队」优先级上升。

## 推荐的桌面端顺序

桌面端可以保留筛选能力，但不要让一排 `0` 成为主视觉。

建议：

- 左侧或顶部仍保留筛选。
- 主区域先显示推荐和行动模块。
- 真实数据为空时，筛选结果区域改成「可加入机会」而不是纯空态。
- 三个原有卡片可以保留，但降级为次要行动，不作为唯一内容。

## 可用数据来源讨论

不需要一开始就做复杂推荐系统，第一版可以使用现有数据：

- `Activity` 中即将开始、公开、未结束的活动。
- `PublicEvent` 中适合组队的公开活动。
- 已有公开组局。
- 当前用户城市、语言、兴趣，如果资料中已有。
- 当前用户是否刚注册、是否有报名、是否有发起、是否有好友。

推荐逻辑可以先规则化：

```text
优先级 1：真实公开局
优先级 2：即将开始且适合组队的公开活动
优先级 3：可一键发起的组队模板
优先级 4：资料补全任务
```

## 不建议的方向

### 不建议只把文案写得更热情

如果底层仍然只有三个跳转卡片，文案再好也不能解决大厅空的问题。

### 不建议用纯营销式 welcome hero

组队大厅是工具型首页，不是落地页。大标题和宣传语不能替代真实内容和行动。

### 不建议默认展示所有活动

那会让组队大厅和活动页重叠。大厅应该把活动包装成「一起去」的机会。

### 不建议一开始就做重匹配算法

冷启动阶段更重要的是规则化推荐和启动动作。复杂算法可以等数据变多后再做。

## 可以分阶段实现

### P0：先解决「看起来空」

目标：不改复杂数据模型，也不引入匹配系统。

可以做：

- 替换当前空态主视觉。
- 在空态中展示公开活动组队机会。
- 加入一键发布找搭子 prompt。
- 保留加好友、发现活动、发起组局，但作为次级入口。

验收标准：

- 新用户进入大厅时不再看到一个大面积空白框。
- 至少能看到 3 到 5 个真实可参与或可发起的机会。
- 用户可以在大厅内完成第一步行动，而不是只能跳走。

### P1：让少数据用户有个性化入口

可以做：

- 根据城市和兴趣筛活动。
- 根据用户报名或收藏历史推荐类似活动。
- 展示「围绕这个活动发起组局」。
- 资料完成度驱动推荐质量。

验收标准：

- 用户有少量数据后，大厅内容开始变得和自己相关。
- 公开活动不再只是列表，而是可组队机会。

### P2：社交推荐和动态流

可以做：

- 同城同兴趣用户推荐。
- 好友参加和好友发起动态。
- 近期活跃搭子。
- 根据活动和兴趣推荐可能一起去的人。

验收标准：

- 活跃用户看到的是自己的社交网络和活动状态。
- 新用户看到的是可启动的社交机会。

## 需要进一步确认的问题

- 新用户是否已有城市字段，还是需要从定位、语言或默认城市推断？
- 公开活动数量不足时，是否允许展示平台模板？
- 是否希望大厅直接暴露用户推荐，还是先用「人群入口」规避隐私和质量风险？
- 「找搭子」是否应该独立于活动存在，还是必须绑定一个活动？
- 移动端大厅是否应完全取代 `/home` 的功能，还是只承担社交行动入口？

## 建议结论

组队大厅新用户空态不应该继续作为传统 empty state 处理。它应该被当作移动端首页冷启动问题来解决。

推荐方向是：

> 用真实公开活动和公开组局填充内容，用一键找搭子降低行动门槛，用资料完成度和兴趣选择提升后续推荐质量。

第一版不需要做复杂匹配系统。只要把当前「空态 + 三个跳转」升级为「真实机会 + 快速行动 + 渐进引导」，就能明显改善新用户进入大厅时的产品感。
