# Next Fun Club 首次团队会议说明书

## 1. 会议目标

本次会议的目标不是讨论所有未来功能，而是让团队对第一阶段开发形成统一认知：

- 项目要解决什么问题
- MVP 第一版具体做什么
- 哪些功能暂时不做
- 技术栈和开发环境如何统一
- 团队成员如何分工
- 第一阶段开发时间如何安排
- 下次会议前每个人需要交付什么

会议结束后，应产出明确的 MVP 范围、第一周任务分配、开发环境要求和初步上线时间。

## 2. 项目背景

Next Fun Club 是一个面向海外中文用户的活动组织与搭子平台。项目首发市场为法国巴黎，首发界面使用中文。

产品希望解决两个核心问题：

1. 用户想参加本地活动，但不知道哪里有人一起玩。
2. 用户想发起饭局、看展、桌游、密室、短途旅行等活动，但缺少结构化的发布和报名工具。

第一阶段产品不做大型票务平台，也不做完整旅行攻略社区，而是先完成一个清晰的组局闭环：

> 用户可以发现活动、发起活动、报名参加活动，并管理自己发起和参与的活动。

## 3. 产品定位

项目名称：Next Fun Club

当前项目代号：chill-club

首发市场：法国巴黎

首发用户：在法国生活、学习、工作、旅行的中文用户

首发语言：中文

核心场景：

- 本地组局：饭局、桌游、看展、密室、运动、City walk
- 旅游搭子：巴黎周边游、欧洲短途、假期结伴旅行

核心 slogan：

> What's next? Fun begins.
>
> 下一场，Fun 开场。

## 4. MVP 核心目标

MVP 的目标不是做完整平台，而是验证以下假设：

1. 巴黎中文用户是否愿意在平台上发布活动。
2. 用户是否愿意报名参加别人发起的活动。
3. 活动卡片、详情页、报名机制是否足够支撑一次真实组局。
4. 项目后续是否有机会接入商家合作、推广、核销和商业化。

第一阶段 MVP 的成功标准：

- 用户可以注册和登录。
- 用户可以浏览活动列表。
- 用户可以查看活动详情。
- 用户可以创建活动。
- 用户可以报名参加活动。
- 用户可以取消报名。
- 用户可以查看自己发起和参与的活动。
- 项目负责人可以手动运营首批活动内容。
- 产品可以部署到线上，供小范围用户试用。

## 5. MVP 功能范围

### 必须完成

以下功能属于 MVP 第一版必须完成的核心闭环：

- 用户注册 / 登录
- 首页活动列表
- 活动详情页
- 创建活动
- 编辑自己发起的活动
- 删除或取消自己发起的活动
- 报名参加活动
- 取消报名
- 活动人数统计
- 活动状态展示：招募中、已满、已结束、已取消
- 个人空间：我发起的活动、我参与的活动
- 移动端基础适配

### 可以简化

搜索功能：

- 第一版只做关键词搜索或分类筛选。
- 不接入 Meilisearch / Elasticsearch。

地图功能：

- 第一版先使用文字地址。
- Mapbox 地图选点放到后续版本。

评论功能：

- 第一版可以暂缓。
- 如果开发进度顺利，再加入基础评论区。

报名审核：

- 第一版先做免审核报名。
- 需审核报名放到第二阶段。

多语言：

- 第一版只交付中文 UI。
- 代码结构可以为 i18n 预留空间。

### 暂时不做

以下功能不进入 MVP 第一版：

- 商家入驻
- 商家后台
- Stripe 支付
- 推广套餐
- CPA 核销
- 智能链接导入
- 活动群聊
- 复杂通知系统
- 推荐算法
- 多城市运营
- 多语言完整切换

## 6. 技术方案

MVP 阶段建议采用单体全栈方案，降低沟通、部署和维护成本。

推荐技术栈：

| 层级 | 技术 |
| --- | --- |
| 语言 | TypeScript |
| 前端 | Next.js App Router |
| UI | Tailwind CSS + shadcn/ui |
| 后端 | Next.js API Routes / Server Actions |
| 数据库 | PostgreSQL |
| ORM | Prisma |
| 认证 | Auth.js 或 Clerk |
| 图片 | Cloudinary，或第一版先使用外链 / 占位图 |
| 部署 | Vercel |
| 数据库托管 | Neon 或 Supabase |

MVP 阶段不建议拆独立后端。当前业务复杂度可以由 Next.js 单体项目支撑。等到活动群聊、后台任务、商家系统、支付和通知系统变复杂后，再考虑拆分 API 服务。

## 7. 开发环境要求

每位开发成员需要安装：

- Node.js LTS
- pnpm
- Git
- VS Code
- PostgreSQL 本地环境，或使用统一远程开发数据库

团队需要统一：

- Node.js 版本
- 包管理器，建议统一使用 pnpm
- 代码格式化方式
- 分支命名规则
- 提交信息格式
- Pull Request 流程
- 环境变量管理方式

项目后续应补充：

```text
.env.example
CONTRIBUTING.md
docs/
```

建议 `.env.example` 初步包含：

```bash
DATABASE_URL=
NEXTAUTH_SECRET=
NEXTAUTH_URL=
CLOUDINARY_CLOUD_NAME=
CLOUDINARY_API_KEY=
CLOUDINARY_API_SECRET=
```

## 8. 建议项目结构

```text
chill-club/
├── apps/
│   └── web/                 # Next.js 页面 + API
├── packages/
│   ├── database/            # Prisma schema + client
│   └── shared/              # 共享类型、常量
├── prisma/
│   └── schema.prisma
├── docs/
│   └── first-team-meeting.md
└── README.md
```

如果团队规模较小，也可以先使用更简单的结构：

```text
chill-club/
├── app/
├── components/
├── lib/
├── prisma/
├── docs/
└── README.md
```

建议第一阶段优先选择简单结构，等项目模块变多后再升级为 monorepo。

## 10. 第一阶段任务拆分

### Sprint 0：项目初始化

目标：让所有开发成员可以在本地运行项目。

任务：

- 初始化 Next.js 项目
- 配置 TypeScript
- 配置 Tailwind CSS
- 配置 shadcn/ui
- 配置 Prisma
- 建立 PostgreSQL 数据库
- 添加 `.env.example`
- 添加基础开发说明
- 建立 Git 分支规则
- 部署一个空页面到 Vercel

交付物：

- 本地可以运行
- 线上可以访问
- 数据库可以连接
- 团队成员开发环境一致

### Sprint 1：活动核心模型

目标：完成活动数据结构和基础页面。

任务：

- 设计 Prisma schema
- 创建 User、Activity、Category、Participation 表
- 创建活动列表页
- 创建活动详情页
- 创建活动发布页
- 添加基础分类数据
- 添加 seed 数据

交付物：

- 可以看到活动列表
- 可以进入活动详情
- 可以创建一条活动

### Sprint 2：用户和报名闭环

目标：完成登录和报名流程。

任务：

- 用户注册 / 登录
- 获取当前用户状态
- 报名参加活动
- 取消报名
- 活动人数统计
- 活动满员限制
- 权限限制：只能编辑自己的活动

交付物：

- 用户可以登录
- 用户可以报名
- 用户可以取消报名
- 活动人数可以正确更新
- 满员活动不能继续报名

### Sprint 3：个人空间和上线准备

目标：完成 MVP 可试用版本。

任务：

- 我的发起活动
- 我的参与活动
- 编辑活动
- 取消活动
- 移动端适配
- 错误状态
- 空状态
- 基础 QA
- 生产环境部署
- 准备首批测试活动

交付物：

- 小范围用户可以真实试用
- 项目负责人可以运营首批内容
- 可以收集反馈进入下一阶段

## 11. 时间计划建议

如果团队规模为 2 到 4 人，建议第一版控制在 4 周内。

### 第 1 周

重点：项目初始化、开发环境、数据库、基础 UI。

目标：

- 项目可以本地运行

### 第 2 周

目标：

- 可以创建活动
- 可以编辑活动
- 可以展示活动详情
- 活动卡片字段基本完整

### 第 3 周

重点：用户系统和报名流程。

目标：

- 用户可以登录
- 用户可以报名

## 12. 开发协作规则

建议采用以下规则：

- `main` 分支保持可部署状态。
- 每个功能开独立分支。
- 合并前必须本地运行通过。
- 不提交 `.env` 文件。
- 数据库结构修改必须通过 Prisma migration。
- 每个 PR 说明做了什么、如何测试。
- UI 组件尽量复用。
- 功能优先，复杂优化后置。

分支命名示例：

```bash
feature/activity-list
feature/create-activity
feature/auth
fix/join-activity-count
```

提交信息示例：

```bash
feat: add activity list page
feat: add join activity flow
fix: prevent joining full activity
chore: setup prisma
```

## 15. 当前主要风险

### 功能范围过大

当前策划书覆盖 C 端、B 端、旅行、商家、支付、搜索、地图、智能导入等多个方向。方向成立，但 MVP 必须收窄。

建议第一阶段只验证 C 端组局闭环。

### 冷启动难度

这类平台早期最大问题不是技术，而是没有活动和没有人报名。

建议项目负责人手动准备第一批活动，例如：

- 巴黎周末 Brunch
- 看展搭子
- 桌游局
- 密室逃脱组队
- City walk
- 周边一日游

### 信任机制不足

陌生人组局需要基础信任。

MVP 后应尽快补充：

- 头像和个人简介
- 发起人历史活动
- 报名留言
- 活动评论
- 举报机制

### 商业化不宜过早

商家功能可以作为长期方向，但第一版不应拖慢 C 端闭环。

建议等平台有真实活动、真实报名和基础用户反馈后，再推进商家入驻、推广位和核销能力。
