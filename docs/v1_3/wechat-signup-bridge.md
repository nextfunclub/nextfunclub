# 微信报名桥用户分析、可行性讨论与落地方案

对应规划分支：

```text
docs/wechat-signup-bridge-planning
```

## 背景

当前微信群里的活动组织方式主要依赖：

- 一大段活动说明文字
- 群内手动接龙
- 手动统计报名人
- 手动整理到 Excel 或其他表格

这种方式的优点是门槛低、传播快、符合微信群习惯；缺点是数据无法沉淀到 Next Fun，人数、报名状态、取消、候补、邀请来源和后续社交关系都需要人工维护。

本方案的目标不是立刻把微信账号体系、App、小程序和 Next Fun 全部打通，而是先验证一个更现实的桥接路径：

```text
微信群 / 微信私聊 / 海报二维码
  -> 公开 H5 活动页
  -> 游客免登录报名
  -> 报名成功后生成可分享内容
  -> 邀请和报名数据回流 Next Fun
  -> 可选绑定现有账号或后续认领
```

## 用户原始设想

1. 发起人需要登录网站发起组队，或者根据活动建立组队。
2. 发起人或参与人可以分享当前活动的分享图 / 链接到微信。
3. 任意参与者点击“接龙”按钮后，将活动卡片转发到微信某个人或群里。
4. 发送内容是一张微信展示卡，包含活动标题、内容、当前报名人头像和参与人数。
5. 新用户点击卡片查看详情并接龙，接龙后继续回到转发动作，形成循环。
6. 只有分享成功后才算接龙成功。
7. 微信 WebView 中 Google 登录不可用，因此游客也应该可以直接参加；之后可以关联已有账号。

## 初步可行性结论

整体方向可行，但第一版需要调整“接龙成功”的定义。

可行部分：

- 从微信群打开公开 H5 活动页。
- 在微信 WebView 内免登录报名。
- 报名后生成报名凭证、邀请链接、群文案和活动海报。
- 使用邀请 token 追踪谁带来了报名。
- 游客报名后再绑定 Next Fun 账号或后续认领。
- 点击微信卡片后进入最新的活动详情页，页面展示最新报名人数和头像。

不适合作为第一版硬依赖的部分：

- H5 页面通常无法可靠判断用户是否真的把内容发给了某个微信好友或微信群。
- H5 页面通常拿不到用户选择了哪个群、哪个好友。
- 已经发送到微信群里的微信卡片不应假设可以实时动态刷新头像和人数。
- 微信登录、App 拉起、小程序授权都依赖额外主体、审核和平台能力，不适合现在作为主路径。

因此第一版建议把闭环定义为：

```text
报名成功 = 接龙成功
分享成功 = 传播增强
邀请带来新报名 = 可追踪转化
```

也就是说，网站内数据先以“报名成功”作为可信事实，微信转发作为传播渠道，不把“是否真的分享到某个群”作为唯一成功条件。

## 核心产品判断

### 1. 先做微信报名数据，不先做微信账号打通

当前阶段最重要的是让微信群用户能完成报名，而不是先强制完成微信登录、Google 登录、App 安装或小程序授权。

推荐主路径：

```text
用户在微信群看到活动
  -> 点链接 / 扫码
  -> 打开公开 H5 活动页
  -> 填昵称、联系方式、人数、备注
  -> 报名成功
  -> 生成可复制群文案 / 邀请海报 / 报名凭证
```

### 2. 微信端承担短链路，Next Fun 承担数据沉淀

微信负责：

- 传播入口
- 群内曝光
- 私聊 / 群聊转发
- 临时活动通知入口

Next Fun 负责：

- 活动详情
- 报名表单
- 名额、候补、取消、导出
- 邀请归因
- 游客身份
- 后续账号绑定
- 好友关系和活动社交沉淀

### 3. App 和小程序是增强路径，不是报名门槛

当前条件是：只有浏览器网站和移动端适配网页，没有 App，也没有小程序。

当前不建议设计成：

```text
微信群 -> 下载 App -> 注册 -> 找活动 -> 报名
```

建议设计成：

```text
微信群 -> H5 报名成功 -> 可选下载 App / 绑定账号 / 查看好友活动
```

因此当前大项优化的判断标准应该是：

- 只要能通过 H5 完成，就放入近期优化 list。
- 需要 App 上架、Universal Links、App 内路由的内容，全部放入远期。
- 需要小程序主体、备案、审核或微信开放平台能力的内容，全部放入远期验证。
- 需要微信登录才能成立的内容，不进入第一阶段主链路。
- 需要判断用户是否真的分享到微信群的内容，不作为第一阶段成功条件。

## 当前只有 H5 网站时的优化 list 判断

### 近期应该做

这些内容只依赖现有 Web / H5 能力，适合进入当前或下一阶段开发：

1. 公开 H5 活动页

- 微信 WebView、手机浏览器和桌面浏览器都能打开。
- 不要求登录。
- 不触发 Google 登录、Clerk 登录墙或无法返回的授权跳转。

2. 游客免登录报名

- 用户只填最少字段即可参加。
- 报名记录允许没有正式账号。
- 报名成功后生成凭证和后续认领 token。

3. 活动报名状态展示

- 活动页展示当前人数、容量、候补状态。
- 展示报名人头像或首字符头像。
- 头像过多时显示 `+N`。

4. 微信传播素材

- 复制微信群文案。
- 生成活动海报二维码。
- 生成个人邀请链接。
- 生成个人邀请海报。

5. 邀请归因

- 分享链接携带 `shareToken`。
- 记录点击来源。
- 新报名关联邀请来源。
- 发起人能看到不同分享带来的报名数。

6. 报名管理

- 发起人可查看报名名单。
- 支持取消、候补、备注。
- 支持导出 CSV / Excel。

7. 游客后续认领

- 报名成功页提供绑定账号入口。
- 登录后能把游客报名合并到现有账号。
- 不把绑定作为报名门槛。

### 需要先验证再做

这些内容可能可做，但需要先做技术或产品验证：

- 微信内置浏览器 Cookie、localStorage、返回路径是否稳定。
- 微信分享卡片的标题、描述、缩略图缓存策略。
- 是否具备接入微信 JS-SDK 的主体、域名、签名和安全域名条件。
- 分享回调能否稳定触发，以及是否符合产品预期。
- 联系方式收集应该使用手机号、微信号还是只用昵称加备注。
- 游客取消报名如何确认本人身份。
- 海报图动态生成服务的性能、缓存和成本。

### 暂不进入当前优化 list

这些内容依赖 App、小程序或更重的平台能力，不适合当前阶段：

- App 拉起。
- App 内活动详情页。
- 安装后恢复活动上下文。
- Universal Links / App Links。
- 微信开放平台 App 登录。
- 小程序报名页。
- 小程序 URL Link / URL Scheme。
- 微信好友关系读取或微信群成员读取。
- 完整陌生人 P2P 私聊。
- 依赖“分享成功后才算接龙成功”的强流程。

### 当前阶段推荐优先级

在只有 H5 的前提下，优先级建议调整为：

```text
H5 公开报名页
  -> 游客报名
  -> 报名凭证
  -> 群文案 / 海报
  -> 邀请归因
  -> 发起人名单管理
  -> 游客认领账号
  -> 活动内轻互动
```

也就是说，当前的大项优化不应该叫“微信 App 桥”或“微信账号打通”，更准确的名称是：

```text
微信 H5 报名桥
```

## 关键用户链路

### 链路 A：发起人创建并分享到微信

```text
登录 Next Fun
  -> 创建组队或从活动发起组队
  -> 生成公开报名页
  -> 生成邀请链接 / 邀请海报 / 群文案
  -> 分享到微信群或微信好友
```

页面和数据要求：

- 活动必须有公开 H5 落地页。
- 分享链接必须携带 `shareToken` 或 `inviterToken`。
- 发起人可以查看报名列表、候补、来源和导出。
- 分享素材要适配微信卡片和海报二维码。

### 链路 B：微信群用户免登录报名

```text
微信中点击活动链接
  -> 打开公开 H5 活动页
  -> 查看时间、地点、费用、人数、发起人
  -> 点击我要参加
  -> 填写最少字段
  -> 报名成功
  -> 获得报名凭证
```

最少字段建议：

- 昵称
- 联系方式，优先手机号或微信号二选一，具体合规后确认
- 报名人数
- 备注，可选

报名成功后展示：

- 报名凭证号
- 当前活动人数
- 复制群文案
- 生成我的邀请海报
- 查看报名人
- 可选绑定 Next Fun 账号

### 链路 C：参与者继续邀请别人

```text
已报名用户
  -> 点击邀请 / 接龙
  -> 生成个人邀请链接和海报
  -> 复制文案或转发
  -> 新用户通过该链接报名
  -> 系统记录邀请来源
```

这里的“接龙”建议定义为：

- 用户自己完成报名后已经进入接龙。
- 用户继续转发是“帮忙邀请”。
- 新用户通过这个邀请链接报名后，才形成可验证转化。

### 链路 D：游客报名绑定账号

```text
游客报名成功
  -> 获得 claimToken
  -> 之后登录或注册 Next Fun
  -> 验证手机号 / 邮箱 / 一次性 token
  -> 将游客报名合并到正式账号
```

目标：

- 不阻挡首次报名。
- 不丢失游客报名数据。
- 后续可以沉淀到好友关系、活动历史和消息系统。

## 路由规划

建议新增或预留：

| 路由 | 登录要求 | 作用 |
| --- | --- | --- |
| `/e/[eventSlug]` | 不需要 | 微信公开活动落地页 |
| `/join/[shareToken]` | 不需要 | 带邀请关系的报名入口 |
| `/r/[registrationToken]` | 不需要 | 报名凭证页 |
| `/claim/[claimToken]` | 可选登录 | 游客报名绑定账号 |
| `/wechat/events/[eventId]/share` | 可选登录 | 生成微信分享素材 |
| `/admin/events/[eventId]/registrations` | 需要登录 | 报名名单、导出和管理 |

公开路由需要从登录保护中排除，避免微信 WebView 中出现 Google 登录或 Clerk 握手阻断。

## 分享与微信卡片策略

### 微信卡片形式的可行性补充

用户期望的不是“文字 + 裸链接”，而是微信群里常见的卡片形式。这里需要拆成四类能力判断：

| 形式 | 当前是否适合主路径 | 说明 |
| --- | --- | --- |
| 普通 H5 链接卡片 | 可作为目标，但需要验证 | 用户从微信内分享网页时，微信可能展示网页标题、描述和缩略图；但如果没有 JS-SDK 配置，展示效果和缓存不可完全控制 |
| 自定义 H5 分享卡片 | 需要公众号 / JS-SDK 条件 | 通常需要公众号 `appId`、JS 接口安全域名、服务端签名和 `wx.config`，再配置分享标题、描述、图片和链接 |
| 小程序卡片 | 当前不适合 | 需要小程序主体、类目、备案、审核和小程序页面，不符合当前“只有 H5 网站”的阶段 |
| 收藏后再转发形成卡片 | 不适合主流程 | 用户路径太绕，无法稳定控制样式，也难以统计，不适合作为产品主链路 |

因此当前阶段的判断是：

- [ ] 可以把“微信卡片展示效果”作为 P0/P1 的技术验证项。
- [ ] 不把“必须发出微信卡片”作为游客报名第一版的前置条件。
- [ ] 不把“分享成功后才算接龙成功”作为第一版主逻辑。
- [ ] 如果没有公众号 / 小程序条件，先用 H5 页面 + 群文案 + 海报二维码 + 邀请链接完成闭环。
- [ ] 如果后续具备公众号条件，再接入 JS-SDK 做自定义卡片。

### 如果要做 H5 自定义分享卡片

目标效果：

```text
微信群里的链接卡片
- 标题：活动标题
- 描述：时间、地点、当前人数
- 缩略图：活动封面 / 动态活动海报
- 链接：公开 H5 活动页，带 shareToken
```

大致需要：

- 公众号或可用的微信 JS-SDK `appId`
- 微信公众平台配置 JS 接口安全域名
- 服务端获取并缓存 `jsapi_ticket`
- 服务端按当前 URL 生成 `signature`
- H5 页面加载微信 JS-SDK
- 前端 `wx.config`
- 前端配置发送给朋友 / 分享到朋友圈的标题、描述、图片和链接
- 分享链接中保留 `shareToken`，用于邀请归因

需要注意：

- H5 不能自动替用户选择微信群或好友。
- 即使有分享接口，也不应依赖它判断用户发给了谁。
- 分享回调和目标群信息不能作为稳定业务事实。
- 微信可能缓存链接卡片信息，动态人数和头像更适合在用户点进 H5 后展示。

### 当前 H5 阶段的推荐降级方案

在没有公众号和小程序前，推荐做三种分享入口：

1. 复制群文案

```text
周六羽毛球，20 人封顶，目前 12 人报名。
时间：6 月 13 日 19:00
地点：XX 体育馆
点这里报名：短链接
```

2. 生成活动海报

- 图片里包含标题、时间、地点、人数、发起人和二维码。
- 用户保存图片后发微信群。
- 这是最稳定的“视觉卡片”替代方案。

3. 生成个人邀请链接

- 链接带 `shareToken`。
- 如果微信能展示为链接卡片，就使用基础网页 metadata。
- 如果不能，也能通过海报二维码和文案完成报名转化。

第一版产品文案可以避免承诺“发送微信卡片”，改成：

```text
分享活动
- 复制群文案
- 生成邀请海报
- 复制报名链接
```

后续具备公众号条件后，再升级为：

```text
分享到微信卡片
```

### 无公众号条件下的卡片实现穷举

结论：如果目标是“稳定、可控地生成微信聊天里的图文卡片，并且卡片标题、描述、缩略图都由 Next Fun 控制”，在没有公众号 / 小程序 / 微信开放平台主体能力时，基本没有完全正规的主路径。能做的是把可行方案分成“可生产使用”“可实验验证”“不建议依赖”三类。

| 方案 | 是否需要公众号 / 小程序 | 能否接近卡片 | 是否建议 |
| --- | --- | --- | --- |
| H5 页面加完整 metadata / Open Graph / 动态海报图 | 不需要 | 可能形成基础链接卡片，但不可控 | 建议作为 P0 实验 |
| 用户在微信里打开 H5 后点右上角分享 | 不一定 | 可能比直接粘贴链接更容易出现卡片，但不可控 | 建议测试 |
| 海报图片 + 二维码 | 不需要 | 视觉上最像卡片，但不是可点击链接卡片 | 建议作为第一版主方案 |
| 复制群文案 + 短链接 | 不需要 | 不是卡片，但转化稳定 | 建议保底 |
| 收藏网页后从微信收藏转发 | 不需要 | 有机会形成卡片 | 不建议作为主流程，步骤太绕 |
| 扫二维码进入页面后再分享 | 不需要 | 有机会形成卡片 | 可实验，不保证 |
| 第三方短链 / H5 营销平台中转 | 不需要自己有公众号 | 可能形成卡片 | 可小流量测试，依赖和风控风险高 |
| 合作方公众号菜单 / 文章 / JS-SDK | 需要合作方公众号 | 可以做可控卡片 | 可作为商业合作路线 |
| 自己注册公众号接 JS-SDK | 需要主体和认证 | 可以做可控卡片 | 当前做不了 |
| 小程序卡片 | 需要小程序 | 可以做小程序卡片 | 当前做不了 |

#### 方案 1：H5 metadata 最小实验

给公开活动页补齐：

```text
title
description
og:title
og:description
og:image
og:url
og:type
twitter:card
twitter:title
twitter:description
twitter:image
```

同时为每个活动生成一张稳定尺寸的分享图：

```text
/api/public/events/[eventId]/share-image.png
```

分享图内容包括：

- 活动标题
- 时间地点
- 当前人数
- 发起人
- 报名二维码

注意：

- 这不能保证微信一定展示为卡片。
- 即使展示为卡片，微信也可能缓存旧标题、旧缩略图。
- 动态人数和头像不应依赖微信聊天卡片刷新，应该让用户点进 H5 后看最新数据。

#### 方案 2：微信内右上角分享实验

测试路径：

```text
微信群点击链接
  -> 打开 H5 活动页
  -> 用户点右上角 ...
  -> 发送给朋友 / 分享到群
```

需要记录：

- 是否展示为卡片
- 标题是否取网页 title
- 描述是否取 meta description
- 缩略图是否取 og:image
- 不同入口是否不同：直接链接、二维码扫码、收藏入口、浏览器打开再回微信

这个方案不需要开发公众号，但结果不可控。可以作为实测项，不作为产品承诺。

#### 方案 3：海报二维码作为视觉卡片主方案

如果没有公众号，这是最稳的“类卡片”方案：

```text
生成活动海报
  -> 用户保存或复制图片
  -> 发到微信群
  -> 群成员长按识别二维码 / 扫码
  -> 打开 H5 报名页
```

优点：

- 不依赖公众号。
- 展示完全可控。
- 可以放报名人数、头像、二维码、发起人。
- 适合微信群传播。

缺点：

- 图片本身不是可点击链接。
- 需要用户长按识别二维码。
- 转化链路比点击卡片长一步。

因此第一版建议把按钮命名为：

```text
生成微信海报
```

而不是：

```text
发送微信卡片
```

#### 方案 4：第三方短链 / 营销 H5 平台中转

有些第三方服务可能自己具备公众号或微信分享能力，能生成一个可控卡片，再跳转到 Next Fun。

潜在流程：

```text
Next Fun 活动页
  -> 调用第三方生成短链或落地页
  -> 微信中分享第三方短链卡片
  -> 用户点击后跳转 Next Fun H5 报名页
```

风险：

- 品牌和域名不完全属于 Next Fun。
- 数据链路依赖第三方。
- 可能收费。
- 可能被微信风控或限制。
- 如果第三方落地页中转太明显，用户信任感下降。

判断：

- 可以用一个真实活动做小范围 A/B 测试。
- 不建议作为长期核心能力，除非服务稳定、合规、可导出数据。

#### 方案 5：借合作方公众号能力

如果自己无法注册认证公众号，可以和有主体的活动组织方、俱乐部、场馆合作。

可能方式：

- 合作方公众号配置 JS 接口安全域名。
- 用合作方公众号文章或菜单入口承载活动链接。
- Next Fun 提供 H5 报名页和后台数据。
- 分享卡片由合作方公众号能力完成。

风险：

- 域名、安全域名、品牌归属和数据权限要谈清楚。
- 合作方公众号主体需要信任 Next Fun 的页面。
- 多合作方场景会带来配置和维护成本。

判断：

- 如果未来有稳定线下组织方，这是最接近正规可控卡片的非自有主体路径。
- 不适合作为第一个原型的前置条件。

#### 推荐实验顺序

```text
1. 给活动页补齐 metadata 和动态分享图
2. 用真实微信测试 6 种入口：
   - 直接粘贴链接
   - 微信内打开后右上角分享
   - 二维码扫码进入后分享
   - 收藏后转发
   - 短链接后分享
   - 第三方短链 / H5 平台
3. 记录卡片展示、标题、描述、缩略图、缓存和点击转化
4. 如果无公众号卡片不可控，则把“微信海报”作为主方案
5. 后续再评估合作方公众号或自有主体
```

实验验收表：

| 测试项 | 结果 |
| --- | --- |
| 是否生成卡片 | 待测 |
| 标题是否可控 | 待测 |
| 描述是否可控 | 待测 |
| 缩略图是否可控 | 待测 |
| 是否缓存旧图 | 待测 |
| 是否保留 shareToken | 待测 |
| 用户是否愿意使用 | 待测 |

当前产品结论：

- 第一版主按钮：`生成微信海报`
- 第一版辅助按钮：`复制报名链接`、`复制群文案`
- 技术实验按钮或内部工具：`测试微信卡片`
- 暂不对用户承诺：`发送微信卡片`

### 第一版分享素材

- 群文案：最稳定，用户复制后发群。
- 活动海报：包含二维码，适合转发群和朋友圈。
- 个人邀请海报：二维码带邀请参数，适合追踪转化。
- 普通链接卡片：用于微信聊天中展示标题、描述和缩略图。

### 微信卡片的现实限制

需要单独验证：

- 微信是否缓存当前链接的标题、描述和缩略图。
- 同一 URL 的分享卡片更新频率和缓存策略。
- 是否需要每次分享生成带版本号或 token 的 URL，避免卡片图片缓存。
- H5 是否能调用微信 JS-SDK 分享接口，以及需要的公众号 / JS 安全域名 / 签名配置。
- 是否能获得可靠的分享成功回调；即使有回调，也不应假设能知道目标群或好友。

### 建议实现方式

第一版不要依赖“系统弹窗确认分享成功才算接龙成功”。

推荐：

```text
报名成功
  -> 生成个人 inviteToken
  -> 用户复制文案或保存海报
  -> 后续新报名通过 inviteToken 归因
```

如果未来接入微信 JS-SDK：

- 只把它用于优化分享标题、描述、缩略图。
- 不把 JS-SDK 回调作为报名成功的唯一依据。
- 分享目标和群信息不进入产品假设。

## 报名卡片内容设计

微信中展示的链接卡片或海报建议包含：

- 活动标题
- 时间
- 地点
- 当前人数，例如 `12 / 20`
- 发起人
- 已报名头像，最多展示 5 个
- 超出人数显示 `+N`
- CTA：点击报名 / 查看详情

注意：

- 已发出的微信聊天卡片不应假设能动态刷新。
- 用户点击卡片进入 H5 后，H5 页面必须展示最新人数和头像。
- 如果要让卡片图片尽量新，需要考虑动态 OG 图或海报图，并处理微信缓存。

## 游客报名与账号关联

### 第一版身份策略

游客报名可以使用：

- `guestId`：服务端生成，写入 cookie。
- `registrationToken`：报名凭证 token。
- `claimToken`：未来绑定账号使用。
- `phoneHash`：如果收集手机号，用于去重和后续认领。

报名记录允许：

```text
registration.userId = null
registration.guestId = 有值
```

### 账号绑定策略

游客后续登录后：

- 通过 `claimToken` 绑定单条报名。
- 通过手机号验证合并历史游客报名。
- 合并后保留原 guest 记录，避免审计和追踪丢失。

## 风险与待验证问题

- [ ] 当前 Clerk middleware 是否会拦截公开活动页和报名页
- [ ] 微信 WebView 中 Google 登录、Clerk 跳转、Cookie 写入是否稳定
- [ ] 微信 JS-SDK 分享接口需要哪些主体、域名和签名配置
- [ ] 微信是否缓存分享卡片缩略图，缓存多久
- [ ] 分享成功回调是否可用、是否可靠、是否能覆盖目标场景
- [ ] 是否需要手机号，或能否先用微信号 / 昵称降低隐私压力
- [ ] 手机号、微信号等联系方式的加密、脱敏、删除和导出规则
- [ ] 游客取消报名如何验证本人身份
- [ ] 一个游客报名多人时，头像和人数如何展示
- [ ] 发起人如何防止恶意报名、刷名额和垃圾留言
- [ ] 活动满员后候补和邀请归因如何结算
- [ ] 海报图生成服务的性能和缓存策略
- [ ] 欧盟用户下手机号、邀请归因和聊天内容是否需要额外 GDPR 提示

## 不建议第一阶段做的内容

- [ ] 强制微信登录
- [ ] 强制下载 App 才能报名
- [ ] 强制 Google 登录
- [ ] 依赖小程序作为唯一入口
- [ ] 依赖 H5 判断用户已分享到具体微信群
- [ ] 假设微信群里已发送的卡片能实时动态更新人数和头像
- [ ] 一开始就开放完整陌生人私聊
- [ ] 一开始就读取或依赖微信好友关系

## 推荐第一步验证

先做一个最小闭环原型：

```text
公开活动页
  -> 游客报名
  -> 报名成功页
  -> 复制群文案
  -> 个人邀请链接
  -> 管理端报名列表
```

这个原型能验证：

- 微信 WebView 是否能顺畅打开
- 游客是否愿意不登录直接报名
- 发起人是否愿意用后台名单替代手动 Excel
- 群文案和海报是否能带来新报名
- Next Fun 是否能通过邀请关系沉淀自己的社交图谱
