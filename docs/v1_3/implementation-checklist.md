# v1.3 功能实现清单

## 当前阶段目标

v1.3 的核心目标不是继续堆功能，而是把 v1.2 已经搭好的活动、组队大厅、好友、通知和收藏体验做顺。

当前主线：

```text
更快打开页面 → 更清楚地区分活动和组队 → 更及时收到通知 → 更自然完成收藏、报名、取消和好友互动
```

本版本最重要的问题是“页面等待时间和操作反馈过慢”。本地环境、预览环境和生产环境都需要分别判断：

- 是否是 Vercel 免费版 / 预览环境冷启动导致
- 是否是 Supabase / Prisma 查询过多或串行查询导致
- 是否是页面级 Server Component 数据请求过重导致
- 是否是用户操作没有乐观更新、没有中间状态导致体感慢
- 是否是缓存策略、组件拆分或列表查询结构不合理导致

v1.3 优先解决“体感速度”和“关键操作反馈”，再进入更细的功能增强。

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull origin dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* → dev
```

不要把性能、通知、活动发现、组队权限、收藏数字等功能合并到同一个分支里。每个分支只解决一个明确问题，方便测试、回滚和回退。

## P4：组队大厅与导航结构优化

### 14. 组队大厅扁平列表与状态筛选

建议分支：

```text
feature/lobby-flat-feed-status-filters
```

设计背景：

当前组队大厅在内容较少时会出现多个大框分散页面注意力，左侧有卡片、右侧大片空白，空状态和分类框架显得过重。`待完成 / 已过期` 作为子框架分类也会让大厅结构变复杂，适合上移为顶部状态筛选。

小功能：

- [x] `全部` 直接展示所有可见组队车，作为真正的全局组队大厅
- [x] 顶部筛选统一承载状态分类，不再在子框架里重复拆 `待完成 / 已过期`
- [x] `待完成` 文案改为 `进行中`
- [x] `已过期` 文案改为 `已结束`
- [x] 已结束组队仍在同一列表结构中展示，但使用灰色弱化状态，和活动卡片状态逻辑对齐
- [x] 内容不足时减少大面积空框和嵌套卡片，让组队卡片自然聚合展示
- [x] 空状态只在当前筛选确实没有内容时出现，并且不撑出过大的空白区域
- [x] 保留开放局、私人局、好友相关筛选能力，但避免和状态筛选形成重复框架

验收标准：

- 用户进入组队大厅后先看到组队车列表，而不是多个空的大区块
- `全部` 能展示所有可见组队车
- `进行中 / 已结束` 能在顶部筛选中完成切换
- 已结束组队视觉上明显弱化，但不破坏列表布局
- 网页端和移动端都不会因为空状态出现明显大面积空白

- [x] 移动端头像、倒计时和现有标签不互相挤压

- [x] 个人空间布局需要适配现有 header、底部导航、卡片样式和纸张背景，不引入突兀的大框或营销式 hero
- [x] 个人空间移动端底部导航不遮挡最后一张卡片、空状态按钮或分页/切换控件

验收标准：

- 网页端个人空间在内容少时也有完整页面重心，不再显得页面中间空荡
- 网页端有 0、1、2、多张活动卡片时布局都稳定，不出现奇怪的大空白
- 移动端进入个人空间后，首屏不会被个人资料区完全占满
- 移动端用户能更快看到自己发起、参加或收藏的活动内容
- 好友号、昵称编辑、好友入口和统计信息都保留，但层级更轻
- 自己的个人空间和朋友主页都保持各自合理的信息层级
- 空状态有明确下一步入口，但文案精简，不制造新的大卡片负担
- 新布局与现有网站视觉一致，桌面端、移动端和极窄屏都不出现裁切或重叠

### 26. 后台与搜索页面移动端布局优化

建议分支：

```text
feature/admin-search-mobile-layout-polish
```

设计背景：

当前几个偏工具型页面在移动端的布局还保留了网页端信息块的结构，导致手机上出现大卡片堆叠、指标纵向拉长、首屏信息密度低和操作路径不够清楚的问题。尤其 `/admin/analytics` 的产品信号和运营总览模块，在移动端看起来像一列很长的指标墙；`/admin/reports` 的统计卡片和运营概览同样占用大量纵向空间；`/admin/data-scraper` 作为导入和调试工具，也需要把状态、操作和日志更适合手机扫读。

小功能：

- [x] `/admin/analytics` 移动端顶部说明、tab / 分组导航和关键指标需要压缩，避免首屏只看到导航和少量指标

- [x] `/admin/analytics` 移动端指标卡片不应单列过高堆叠，优先使用双列、紧凑横条或分组摘要
- [x] `/admin/analytics` 图表、表格或排行榜在移动端需要避免横向溢出，必要时改成卡片化摘要

- [x] `/admin/data-scraper` 的主要操作按钮在移动端需要保持可点、清楚，不被底部导航遮挡
- [x] `/admin/data-scraper` 的日志、错误、导入结果在手机上需要可读，长文本需要折行或收起，不制造横向滚动
- [x] `/admin/reports` 移动端顶部统计和运营概览需要压缩，避免大卡片连续占满首屏
- [x] `/admin/reports` 举报列表、处理状态、筛选和操作按钮在手机上要更适合单手浏览和处理
- [x] `/admin/reports` 移动端处理按钮、查看详情、状态徽章不能重叠或被裁切
- [x] `/search?q=巴黎` 移动端搜索结果支持双列展示，优先覆盖活动 / 组局卡片类结果
- [x] 搜索页极窄屏仍保留单列，避免标题、徽章、收藏按钮和操作按钮挤压
- [x] 搜索页不同类型结果（活动、组局、用户、商家等）需要保持清楚分类，不能因为双列而混淆语义
- [x] 搜索结果数量少、无结果、加载态和错误态都需要适配新的移动端布局
- [x] 这些页面的移动端布局需要适配现有 header、底部导航、纸张背景和卡片风格
- [x] 所有涉及移动端双列或紧凑卡片的地方都不能出现横向滚动条

验收标准：

- `/admin/analytics` 移动端首屏能看到核心状态和至少一组有效指标，不再像长指标墙

- 搜索页极窄屏自动回退单列，内容不被挤压

### 27. 移动端好友二维码与扫码添加

建议分支：

```text
feature/mobile-friend-qr-scan-add
```

设计背景：

当前添加好友主要依赖输入好友号或搜索昵称。这个流程在移动端不够顺手：线下见面、微信群聊天或朋友展示手机时，扫码比输入好友号更自然。可以在移动端好友相关入口里增加“我的好友二维码”和“扫码添加好友”，让用户更快建立好友关系。

这个功能可以实现，但需要注意移动端浏览器能力差异：展示二维码本身比较稳定；扫码需要相机权限，最好使用 `getUserMedia` 配合二维码解析库实现，并保留手动输入好友号作为兜底。微信 WebView、iOS Safari、Android Chrome 对相机权限和扫码 API 支持可能不完全一致，因此不能只依赖浏览器原生 `BarcodeDetector`。

小功能：

- [ ] 移动端好友页或添加好友入口增加“我的二维码”，二维码内容优先使用好友号或安全的加好友链接
- [ ] 二维码展示页需要同时显示好友号，方便扫码失败时手动输入
- [ ] 二维码卡片需要适配移动端尺寸，不出现裁切、横向滚动或底部导航遮挡
- [ ] 移动端添加好友入口增加“扫码添加”按钮，桌面端不强行展示扫码入口
- [ ] 扫码功能调用移动端相机权限，用户拒绝权限时给出清楚反馈
- [ ] 扫码识别成功后进入好友确认流程，而不是直接自动添加好友
- [ ] 扫码结果需要校验来源和格式，只接受平台好友号或平台生成的加好友链接
- [ ] 如果扫码结果对应自己，需要提示不能添加自己
- [ ] 如果扫码结果已经是好友或已有待处理申请，需要展示现有关系状态
- [ ] 相机不可用、浏览器不支持或微信 WebView 限制时，需要保留手动输入好友号入口
- [ ] 扫码页面退出、切换路由或关闭弹窗时需要停止摄像头，避免后台继续占用相机
- [ ] 二维码生成和扫码解析不应把用户隐私信息暴露在二维码里，只放必要的好友标识
- [ ] 移动端扫码弹窗或页面需要和现有 header / 底部导航样式一致，不做突兀的大黑屏工具页
- [ ] 添加成功、申请已发送、失败和权限拒绝都需要有明确反馈

验收标准：

- 移动端用户能打开自己的好友二维码，并让别人扫码识别
- 移动端用户能通过扫码进入添加好友确认流程
- 扫码失败、权限拒绝或浏览器不支持时，仍可手动输入好友号添加
- 扫码不会绕过好友确认和已有关系判断
- 扫码后不会误添加自己，也不会重复发送好友申请
- 关闭扫码后摄像头会释放
- 二维码和扫码入口不影响桌面端现有好友管理体验
- 移动端页面不出现横向滚动条、按钮裁切或底部导航遮挡

### 28. 移动端导航、搜索卡片与个人空间分区一致性

建议分支：

```text
feature/mobile-nav-search-card-consistency
```

设计背景：

移动端底部导航中“我的”更像账号入口，但当前个人空间已经承担用户主页、个人活动中心和好友关系入口的角色，使用“主页”更直观。与此同时，活动卡片和组队卡片已经在首页、活动大厅、组队大厅形成了区分：活动信息卡不显示底部报名 / 组队按钮，组队卡保留主操作按钮。搜索页和个人空间收藏需要跟随这套规则，避免用户在不同页面看到同一类卡片却有不同操作。

搜索页本身是全局搜索，不应该把用户找“想去的内容”拆得过碎。“正在组局”里混入活动会造成语义错误，而把活动和组局拆成两个大区又会让结果分散。更合理的方式是：好友单独作为轻量分区展示少量结果；活动信息和组队混合到一个“活动与组局”主列表里，通过卡片样式区分语义。

个人空间在移动端已经有“我的发起 / 我的参与 / 我的收藏”切换，但网页端如果直接连续展示三段内容，当“我的发起”内容很多时，用户很难快速看到参与和收藏。个人空间应该在桌面端和移动端都使用一致的分区切换，并提供时间升降序，让用户能按最近或最早快速查看内容。

小功能：

- [x] 移动端底部导航中文文案从“我的”改为“主页”
- [x] 搜索页活动 / 组队卡片复用统一判断：活动信息卡不显示底部按钮，组队卡保留底部按钮
- [x] 搜索页卡片按钮逻辑和首页、活动大厅、组队大厅保持一致
- [x] 搜索页好友结果单独分区但限量展示，避免好友很多时占满搜索页
- [x] 搜索页将活动信息和组队合并为“活动与组局”主列表，不再使用“正在组局”承载混合结果
- [x] 搜索页公共活动结果转换为统一活动信息卡样式，不再使用另一套不一致的公共活动卡片
- [x] 搜索页合并活动信息结果时按 publicEventId 去重，避免同一个公共活动重复出现
- [x] 个人空间“我的收藏”中混合公共活动和组队时，活动信息卡不显示底部按钮，组队卡保留底部按钮
- [x] 个人空间网页端也展示“我的发起 / 我的参与 / 我的收藏”切换入口
- [x] 个人空间桌面端和移动端都只展示当前选中的分区，避免内容过长时后续分区被挤到很下方
- [x] 个人空间顶部统计顺序调整为“我的发起 / 我的参与 / 关注 / 粉丝 / 好友”，让社交关系顺序更符合用户理解
- [x] 点击顶部“我的发起”统计卡时，下方内容切换到我的发起
- [x] 点击顶部“我的参与”统计卡时，下方内容切换到我的参与
- [x] 顶部“我的发起 / 我的参与”统计卡需要和下方当前分区保持选中态一致
- [x] 关注、粉丝、好友统计改为打开全屏弹窗，不再使用小型下拉浮层
- [x] 关注、粉丝、好友弹窗需要展示标题、数量、说明、完整列表和关闭操作
- [x] 关注列表中保留取消关注操作，并在弹窗中保持可用
- [x] 关注、粉丝、好友弹窗在移动端和桌面端都需要避免内容裁切，列表过长时内部滚动
- [x] 个人空间分区保留数量展示，让用户知道每个分区有多少内容
- [x] 个人空间增加时间排序切换，默认“最近”
- [x] 时间排序支持切换为“最早在前”，覆盖我的发起、我的参与和我的收藏
- [x] 排序按钮只在当前分区内容超过 1 条时展示，避免无意义控件
- [x] 分区切换和排序按钮需要适配移动端宽度，不出现文字裁切或横向滚动
- [x] 保持朋友主页和自己的个人空间语义区分：自己的页面使用“我的发起 / 我的参与 / 我的收藏”

验收标准：

- 移动端底部导航显示“主页”，且入口仍然指向个人空间
- 搜索页里活动信息卡和组队卡的底部按钮表现与首页、活动大厅、组队大厅一致
- 搜索页好友结果不会因数量过多占据大量页面空间
- 搜索页主结果以“活动与组局”混合展示，用户能通过卡片样式快速区分活动信息和组队
- 搜索页公共活动卡片视觉与活动大厅、首页和组队大厅里的活动信息卡保持一致
- 搜索页不会重复展示同一个公共活动
- 个人空间收藏列表中，公共活动不会出现误导性的报名 / 组队主按钮
- 网页端个人空间可以快速在我的发起、我的参与、我的收藏之间切换
- 顶部统计里的我的发起、我的参与能直接联动下方分区
- 顶部统计里的关注、粉丝、好友顺序正确，且点击后打开全屏弹窗
- 关注、粉丝、好友弹窗在列表为空和列表很多时都能清楚展示，不遮挡、不裁切
- 关注弹窗里可以继续取消关注
- 我的发起内容很多时，不会把我的参与和我的收藏挤到难以发现的位置
- 默认排序为最近，用户可切换为最早
- 分区切换、数量、排序按钮在桌面端和移动端都不裁切、不重叠
- 新交互与现有个人空间、卡片和底部导航样式一致

### 29. 移动端卡片错列瀑布流布局探索

建议分支：

```text
feature/mobile-card-masonry-layout
```

设计背景：

移动端活动信息卡和组队卡已经通过样式做了区分，但两类卡片高度天然不同：组队卡通常会多出报名人数、参与者头像和主操作按钮，活动信息卡则更短。如果移动端继续使用严格双列等高行布局，同一行中较短的活动卡下方会留下明显空白，页面密度和视觉节奏都会受到影响。

可以探索类似小红书帖子流的错列瀑布流布局：左右两列不强制等高，让卡片自然向上补位，提高移动端首屏和连续浏览的信息密度。但这个方案只适合浏览型卡片流，不适合所有页面。需要重点验证阅读顺序、分页体验、加载稳定性和性能，确保不会因为“更密”而让用户找不到内容。

最终落地页面：

- 移动端 `/search` 的“活动与组局”主结果流

暂不落地页面：

- 移动端首页“最近活动”，保持普通双列网格
- 移动端 `/lobby` 组队大厅，保持普通双列网格
- 移动端 `/activities` 活动大厅，保持普通双列网格
- 活动详情页、组队详情页等需要线性阅读的信息页
- 个人空间主页，除非后续确认用户更需要浏览密度而不是分区稳定性
- 后台管理页、举报页、数据页等任务型页面
- 任何需要严格时间线、审核顺序或表格对齐的页面

小功能：

- [x] 搜索页移动端“活动与组局”结果流支持错列瀑布流布局
- [x] 首页、组队大厅和活动大厅回退普通双列网格，不使用瀑布流
- [x] 桌面端和平板端保持现有网格或栅格布局，不强行改成瀑布流
- [x] 极窄屏幕仍回退单列，避免卡片宽度过窄导致标题、徽章、按钮和头像挤压
- [x] 使用轻量 React 分列方案，不引入重型瀑布流 JS 库
- [x] 移动端错列按卡片预估高度分配到较短列，减少一侧连续出现高卡片导致的空洞
- [x] 移动端只有 1 条结果时保留单列展示，不强行显示半宽卡片
- [x] 错列布局下活动卡和组队卡仍保留现有视觉区分，不改变卡片语义
- [x] 卡片内徽章、收藏按钮、标题、时间地点、报名人数、头像和主按钮不能重叠或被裁切
- [x] 列间距和卡片间距需要和现有移动端双列风格一致，不出现横向滚动条
- [x] 分页场景下最后一页、奇数数量和少量结果都不能出现突兀的大空白
- [x] 加载态 skeleton、空状态和筛选后少量结果需要适配错列布局
- [x] URL query、筛选、搜索、分页和返回操作不能因为错列布局产生状态错乱
- [x] 图片加载前后需要尽量减少布局跳动，封面图保持稳定比例或占位
- [x] 需要验证视觉阅读顺序：用户从上到下浏览时，不应明显困惑下一张卡片在哪里
- [x] 如果错列导致用户误解时间顺序或分页顺序，需要保留现有双列布局作为回退
- [x] 底部导航、移动端滚动进度条、toast 和浮动按钮不能遮挡最后一行卡片
- [x] 不在任务型后台页面复用错列布局，避免破坏信息处理效率
- [x] 搜索页“活动与组局”不再使用“查看全部活动”跳转按钮，避免组局结果被带到活动页后丢失语义
- [x] 搜索页滚动到底部后自动加载更多活动与组局结果
- [x] 搜索页加载更多失败时提供轻量重试入口
- [x] 搜索页主结果一次展示数量增加，减少用户刚进入搜索页就需要继续加载的频率
- [x] 搜索页高相关结果加载完后，可以继续补充命中部分关键词的低相关度“相关扩展”结果
- [x] 相关扩展结果独立成段展示，不混入高相关主结果，避免用户误解搜索命中权重
- [x] 相关扩展结果继续沿用活动 / 组局卡片语义和移动端错列布局
- [x] 相关扩展结果需要排除完全命中的结果，避免同一内容重复出现
- [x] 搜索页默认不展示已结束的活动 / 组局，减少过期内容干扰当前决策
- [x] 搜索页增加“显示已结束”筛选按钮，用户主动开启后才展示已结束内容
- [x] “显示已结束”状态需要影响首屏结果、滚动加载和相关扩展结果，避免分页前后筛选不一致

验收标准：

- 常见手机宽度下，搜索页活动 / 组局混排列表比严格等高双列更紧凑，减少明显空洞
- 首页、组队大厅和活动大厅仍保持普通网格，不出现瀑布流排布
- 极窄屏幕能稳定退回单列，内容不被挤压
- 卡片点击、收藏、报名 / 查看等操作区域不变小、不误触相邻卡片
- 图片加载、筛选切换、翻页和返回后不出现明显布局跳动
- 用户能通过卡片样式继续快速区分活动信息和组队
- 列表没有横向滚动条，底部导航不遮挡最后一行内容
- 搜索页能在当前页面继续加载更多活动与组局，不跳转到活动页
- 搜索页直接命中的结果优先展示，相关扩展只在后续或主结果不足时出现
- 相关扩展有明确标题说明，不会和完全匹配结果混淆
- 搜索页默认只展示仍可参与或仍有效的活动 / 组局
- 用户点击“显示已结束”后，已结束内容可以被纳入同一搜索结果流
- 如果测试发现阅读顺序明显变差，则不落地瀑布流，保留当前双列布局

### 30. 搜索结果页细节体验补强

建议分支：

```text
feature/search-results-experience-polish
```

设计背景：

搜索结果页已经完成了活动 / 组局混排、默认隐藏已结束、移动端错列布局、滚动加载和相关扩展结果。下一步不需要再大改结构，而是补齐搜索结果页的“解释性”和“可继续探索性”：用户需要知道为什么搜到这些内容、为什么没有看到某类结果、好友很多时怎么继续找、以及相关扩展到底是不是完全匹配。

这组优化应该保持轻量，不把搜索页重新做成复杂筛选页。搜索页的主任务仍然是快速找到内容，细节优化只负责降低困惑和减少误点。

小功能：

- [ ] 好友结果支持“查看更多好友”或展开 / 收起，避免好友很多时只显示少量结果后无法继续查找
- [ ] 好友结果展开后仍保持独立分区，不和活动 / 组局主结果混排
- [ ] 搜索无结果空状态需要更细分：完全无结果、默认隐藏已结束后无当前结果、只有好友 / 商家但无活动组局等场景文案不同
- [ ] 当默认隐藏已结束导致活动 / 组局为空时，空状态需要明确提示可以点击“显示已结束”查看历史结果
- [ ] 活动标题、组局标题、好友昵称、商家名称可以轻量高亮命中的关键词，帮助用户理解匹配原因
- [ ] 关键词高亮不能破坏标题截断、移动端双列 / 错列卡片布局和可读性
- [ ] 相关扩展结果的触发策略继续细化，优先补充城市、主题、活动类型等更有意义的部分匹配
- [ ] 相关扩展结果不无限制下拉，避免低相关内容越补越多，影响用户判断
- [ ] 搜索结果加载中增加轻量 skeleton 或过渡态，减少筛选切换和新搜索时的突兀空白
- [ ] 记录搜索页关键行为数据：点击显示已结束、滚到相关扩展、点击结果、展开好友结果
- [ ] 埋点只记录行为和数量，不记录敏感个人信息或完整隐私内容
- [ ] 移动端和桌面端都保持搜索结果主内容靠前，不因为新增解释文案占据过多首屏空间

验收标准：

- 好友结果很多时，用户能继续查看更多好友，不会被预览数量截断
- 搜索无结果时，用户能明确知道是“真的无结果”还是“当前筛选下无结果”
- 默认隐藏已结束时，如果历史结果可能有内容，页面能自然引导用户打开“显示已结束”
- 关键词高亮能帮助理解匹配原因，但不会让卡片显得杂乱或影响点击
- 相关扩展比当前简单部分匹配更稳，不把明显无关的结果大量推给用户
- 搜索页切换筛选、滚动加载和进入相关扩展时有合理加载反馈
- 新增行为埋点能帮助后续判断搜索体验是否有效改善
- 整体视觉仍和现有搜索页、活动卡片、组局卡片、移动端底部导航风格一致

### 31. 分页组件页码跳转与省略页码优化

建议分支：

```text
feature/pagination-page-jump-ellipsis
```

设计背景：

当前活动大厅和组队大厅分页主要依赖“上一页 / 下一页”和页数摘要。页数较少时问题不大，但当活动总页数很多，例如 192 页时，用户只能连续点击翻页，很难直接到达目标页。分页组件需要升级为更完整但不臃肿的形式：既能保留移动端简洁的上一页 / 下一页，也能在网页端和宽屏移动端提供页码窗口、省略号和页码输入跳转。

分页是浏览效率工具，不应该变成大表单。移动端空间有限，页码能力需要折叠得足够轻：用户能知道当前页、总页数，能快速去首页 / 末页或输入页码，但不会挤占卡片列表空间。

小功能：

- [ ] 活动大厅分页支持页码窗口展示，例如 `1 2 3 ... 192` 或 `1 ... 8 9 10 ... 192`
- [ ] 当前页附近显示有限数量页码，避免总页数很大时铺满一整行
- [ ] 首页、末页、上一页、下一页状态清楚，当前页有明确选中态
- [ ] 页码省略号只表达中间有省略页，不作为可点击控件
- [ ] 支持输入正确页码后跳转到该页
- [ ] 输入页码需要校验范围，小于 1 或大于总页数时不跳转并给出轻量反馈
- [ ] 输入为空、非数字或含非法字符时不改变当前页
- [ ] 移动端分页保持紧凑，优先显示上一页 / 当前页 / 下一页，并提供轻量“跳页”入口
- [ ] 桌面端可以展示更多页码按钮，但不能撑破页面宽度
- [ ] 活动大厅 URL query 分页跳转后需要保留现有筛选、搜索和排序参数
- [ ] 组队大厅客户端分页需要复用同一套交互逻辑，但不强行写入 URL
- [ ] 分页切换后页面位置需要合理，避免用户停留在旧页底部看不到新内容
- [ ] 分页组件需要适配空状态、单页、最后一页、第一页和超大页数
- [ ] 分页样式需要和现有圆角纸张、按钮、进度条视觉保持一致
- [ ] 键盘可访问性需要保留：页码按钮、输入框、确认跳转都能正常聚焦和提交

验收标准：

- 总页数较多时，用户能通过页码窗口理解当前位置和总页数
- 用户能直接输入页码跳转，不需要连续点击几十次下一页
- 页码输入不会破坏筛选、搜索、排序和当前列表状态
- 移动端分页不占用过多纵向空间，也不出现横向滚动条
- 桌面端分页展示更完整，但按钮数量受控
- 首页、末页、第一页、最后一页、单页和非法页码场景都表现稳定
- 活动大厅和组队大厅分页交互一致，但分别适配 URL 分页和客户端分页
