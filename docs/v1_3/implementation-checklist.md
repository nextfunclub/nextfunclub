# v1.3 功能实现清单

## 当前阶段目标

v1.3 的核心目标不是继续堆功能，而是把 v1.2 已经搭好的活动信息、组队大厅、好友、通知和收藏体验做顺。

当前主线：

```text
更快打开页面 → 更清楚地区分活动信息和组队 → 更及时收到通知 → 更自然完成收藏、报名、取消和好友互动
```

本版本最重要的问题是“页面等待时间和操作反馈过慢”。本地环境、预览环境和生产环境都需要分别判断：

- 是否是 Vercel 免费版 / 预览环境冷启动导致
- 是否是 Supabase / Prisma 查询过多或串行查询导致
- 是否是页面级 Server Component 数据请求过重导致
- 是否是用户操作没有乐观更新、没有中间状态导致体感慢
- 是否是缓存策略、组件拆分或列表查询结构不合理导致

v1.3 优先解决“体感速度”和“关键操作反馈”，再进入更细的功能增强。

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull origin dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* → dev
```

不要把性能、通知、活动发现、组队权限、收藏数字等功能合并到同一个分支里。每个分支只解决一个明确问题，方便测试、回滚和回退。

## P0：v1.3 必须优先解决的核心体验

### 1. 性能基线排查与页面流畅度优化

建议分支：

```text
feature/performance-baseline
```

小功能：

- [x] 记录首页、活动发现页、组队大厅、活动详情页、消息页、通知页的加载耗时
- [ ] 区分本地、Vercel Preview、Vercel Production 的耗时差异
- [x] 检查关键页面是否存在串行数据库查询
- [x] 将可并行的数据请求改为并行读取
- [x] 减少列表页不必要的字段查询和重复查询
- [x] 检查 Prisma 查询是否存在 N+1 风险
- [ ] 对非强用户态的公共活动信息读取增加合理缓存或轻量化查询
- [x] 页面切换时保留稳定 loading / shimmer 状态，避免空白等待
- [ ] 输出一份性能结论，明确哪些慢是环境原因，哪些慢是代码原因

验收标准：

- 可以明确说出主要慢点来自哪里
- 活动发现页和组队大厅不应长时间白屏
- 关键页面的数据读取不再明显串行等待
- Preview 环境即使有冷启动，也有明确加载反馈
- `npm run typecheck` 通过

### 2. 耗时操作的乐观更新、Shimmer 和状态反馈

建议分支：

```text
feature/optimistic-ui-feedback
```

小功能：

- [x] 好友申请发送时立即显示“正在发送”，成功后显示“已发送”
- [x] 收藏 / 取消收藏先本地更新状态，失败后回滚
- [x] 搜索提交、创建组队、取消报名、取消组队增加明确处理中状态
- [x] 操作失败时给出清晰反馈，不让用户误以为按钮失效
- [x] 列表加载、搜索结果加载、好友申请加载使用 shimmer 或稳定骨架屏
- [x] 防止用户在同一操作处理中重复点击
- [x] 对移动端按钮状态做适配，避免状态文案换行或挤压

验收标准：

- 用户点击主要按钮后立刻看到状态变化
- 数据库同步失败时 UI 能回滚或提示重试
- 好友申请、收藏、搜索、创建组队不会出现长时间无反馈
- 移动端操作区不被底部导航遮挡
