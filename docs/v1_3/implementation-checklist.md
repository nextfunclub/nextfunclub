# v1.3 功能实现清单

## 当前阶段目标

v1.3 的核心目标不是继续堆功能，而是把 v1.2 已经搭好的活动信息、组队大厅、好友、通知和收藏体验做顺。

当前主线：

```text
更快打开页面 → 更清楚地区分活动信息和组队 → 更及时收到通知 → 更自然完成收藏、报名、取消和好友互动
```

本版本最重要的问题是“页面等待时间和操作反馈过慢”。本地环境、预览环境和生产环境都需要分别判断：

- 是否是 Vercel 免费版 / 预览环境冷启动导致
- 是否是 Supabase / Prisma 查询过多或串行查询导致
- 是否是页面级 Server Component 数据请求过重导致
- 是否是用户操作没有乐观更新、没有中间状态导致体感慢
- 是否是缓存策略、组件拆分或列表查询结构不合理导致

v1.3 优先解决“体感速度”和“关键操作反馈”，再进入更细的功能增强。

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull origin dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* → dev
```

不要把性能、通知、活动发现、组队权限、收藏数字等功能合并到同一个分支里。每个分支只解决一个明确问题，方便测试、回滚和回退。

## P4：组队大厅与导航结构优化

### 14. 组队大厅扁平列表与状态筛选

建议分支：

```text
feature/lobby-flat-feed-status-filters
```

设计背景：

当前组队大厅在内容较少时会出现多个大框分散页面注意力，左侧有卡片、右侧大片空白，空状态和分类框架显得过重。`待完成 / 已过期` 作为子框架分类也会让大厅结构变复杂，适合上移为顶部状态筛选。

小功能：

- [x] `全部` 直接展示所有可见组队车，作为真正的全局组队大厅
- [x] 顶部筛选统一承载状态分类，不再在子框架里重复拆 `待完成 / 已过期`
- [x] `待完成` 文案改为 `进行中`
- [x] `已过期` 文案改为 `已结束`
- [x] 已结束组队仍在同一列表结构中展示，但使用灰色弱化状态，和活动卡片状态逻辑对齐
- [x] 内容不足时减少大面积空框和嵌套卡片，让组队卡片自然聚合展示
- [x] 空状态只在当前筛选确实没有内容时出现，并且不撑出过大的空白区域
- [x] 保留开放局、私人局、好友相关筛选能力，但避免和状态筛选形成重复框架

验收标准：

- 用户进入组队大厅后先看到组队车列表，而不是多个空的大区块
- `全部` 能展示所有可见组队车
- `进行中 / 已结束` 能在顶部筛选中完成切换
- 已结束组队视觉上明显弱化，但不破坏列表布局
- 网页端和移动端都不会因为空状态出现明显大面积空白
