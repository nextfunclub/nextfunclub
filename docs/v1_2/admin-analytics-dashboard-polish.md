# 管理员数据看台优化

建议分支：

```text
feature/admin-analytics-dashboard-polish
```

## 目标

把 `/admin/analytics` 从“数据堆叠页”整理成管理员和运营人员能快速读懂的运营看台。

这个页面优先回答四个问题：

1. 用户有没有在平台上产生有效互动
2. 活动发现是否能带来详情浏览和组队
3. 组队详情是否能带来报名、评论和私聊沟通
4. 举报和社区反馈是否需要管理员及时处理

## 页面原则

- 面向运营人员展示，不使用开发视角的事件名或内部术语。
- 指标优先帮助判断下一步动作，而不是单纯展示数字。
- 网页端保持横向扫描效率，移动端保持从上到下的阅读顺序。
- 不做复杂 BI 系统，不引入额外图表库。
- 不展示用户隐私内容，不展示评论正文、聊天正文、好友号或邮箱。

## 小功能

- [x] 顶部改为运营总览，展示活跃用户、有效互动、报名转化和待处理举报
- [x] 增加优先关注区，提示管理员需要先看哪些运营问题
- [x] 优化活动发现、活动信息转组队、报名转化三个核心漏斗
- [x] 整理沟通、好友和分享指标，减少嵌套卡片
- [x] 将“入口来源”和“公共活动来源价值”合并成独立来源模块
- [x] 将“处理举报”入口放到社区反馈模块内，操作路径更贴近管理员习惯
- [x] 将次级指标改为轻量数字块，降低页面视觉噪音
- [x] 增加 7 天 / 30 天 / 90 天时间范围切换，所有核心指标按同一时间窗口统计
- [x] 增加数据视图切换：总览、用户路径、来源价值、社区反馈，避免指标平铺堆叠
- [x] 总览视图增加使用趋势柱状图，观察发现、组队、报名和沟通的日期变化
- [x] 总览视图增加热门活动排行，按浏览和有效动作综合判断运营重点
- [x] 增加社区反馈模块，展示新增举报、待处理举报、已处理举报和平均处理时间
- [x] 增加举报对象分布，帮助管理员判断风险集中在用户、活动、组队还是评论
- [x] 保留公共活动来源价值表，用于判断哪些来源值得继续维护
- [x] 三语文案改成人可读的运营语言

## 验收标准

- 管理员打开 `/admin/analytics` 后能在首屏判断运营状态
- 页面文案自然、简洁，不出现机械或开发感表达
- 举报压力和来源价值可以在同一页快速看到
- 管理员可以按 7 天、30 天、90 天切换看台数据
- 管理员可以按业务问题切换视图，而不是在同一屏阅读全部指标
- 总览页能看到近期使用趋势和热门活动排行
- 移动端阅读顺序清晰，不依赖横向滚动理解核心信息
- 没有新增数据库结构和新的隐私采集范围
